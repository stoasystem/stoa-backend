# Feature Landscape

**Domain:** Report recovery operations hardening for STOA weekly reports
**Milestone:** v1.6 Report Recovery Operations Hardening
**Researched:** 2026-06-04
**Overall confidence:** HIGH for operator scope and privacy/safety boundaries because findings are grounded in current project, milestone, audit, and runbook artifacts. MEDIUM for implementation complexity because this feature research did not inspect infra source beyond documented CDK/resource constraints.

## Research Scope

v1.6 should harden the existing v1.5 report operations platform for incident-wide work. Existing behavior already includes admin list/detail, single `generation_failed` retry, single resend, selected bulk resend capped at 25, metadata-only responses, production API/live smoke, and an operator runbook. This research focuses only on new v1.6 operator-facing behavior:

- Incident-wide async recovery jobs.
- Immutable audit logs for recovery actions.
- Production admin browser smoke using a real admin session without temporary production admin accounts.
- CI/CD Lambda dist rebuild guard for `stoa-backend/dist` assets used by CDK.

## Anti-Features

Features to explicitly not build in v1.6.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| "Recover all reports" button without filters | Broad incident mutation without operator controls risks emailing or regenerating the wrong customer set. | Require explicit operation type, week/status filters, preflight counts, reason, and confirmation. |
| Incident job that trusts preview eligibility | Report state can change between preview and execution. | Re-read and revalidate every target at execution time. |
| Undo/rollback of completed email sends | Email delivery cannot be unsent, and regeneration may have already changed artifacts/metadata. | Provide cancel/stop for future work and immutable audit of completed changes. |
| Forcing ineligible targets | This would bypass safety semantics from v1.4/v1.5 and risk duplicate sends or regeneration of successful reports. | Mark ineligible targets as `refused` with a reason. |
| Exposing private report artifacts in UI, API, smoke, or audit logs | Project boundary is backend-mediated, metadata-only report operations. Artifact keys, raw HTML/JSON, public URLs, or presigned URLs would violate v1.5 privacy expectations. | Show artifact availability booleans and operational metadata only. Keep artifact reads inside backend recovery services. |
| Direct S3 browser access or presigned artifact links for operators | Increases leakage risk and bypasses backend authorization. | Use admin detail metadata and backend-mediated recovery actions. |
| Temporary production admin account creation for smoke | v1.6 target explicitly says real admin session without temporary production admin accounts. | Use an existing approved admin session/token mechanism, secret-backed CI session, or manual session handoff documented in runbook. |
| Production browser smoke that performs real mutations by default | Routine smoke should not resend emails or regenerate reports. | Keep smoke read-only unless a named non-customer fixture and explicit mutation mode are configured. |
| Step Functions or new AWS services by default | Project constraints say not to invent AWS resources without proving current CDK/resources cannot support the need. | First assess whether existing FastAPI Lambda plus DynamoDB state and chunked invocations can support the async job safely. Add CDK-managed resources only after evidence. |
| Storing audit as mutable fields only | Existing `last_operation` fields are overwritten and cannot prove historical sequence. | Add append-only audit events correlated to report and job IDs. |
| Audit logs containing full request/response blobs | Blobs can accidentally include private artifact markers or unnecessary PII. | Store structured, minimal metadata with explicit allowlisted fields. |
| Local-only dist freshness warning | v1.5 showed stale assets can reach deploy. A warning is too weak. | Make CI/CD and local CDK deploy preflight fail when `dist` is missing or stale. |

## Complexity And Dependencies By Capability

### Async Incident-Wide Recovery

**Operator expectation:** Start a carefully scoped job, see a preview, provide a reason, confirm, monitor progress, stop it if needed, and inspect final per-target results.

**Complexity:** High.

**Core dependencies:**

- Backend service extraction for shared single-target retry/resend behavior.
- Job persistence with stable `job_id`, status, counters, filters, reason, actor, timestamps, and continuation state.
- Per-target result persistence or audit-backed result query.
- Eligibility recheck using the same rules as existing detail/actions.
- UI state for job creation, preview, confirmation, progress polling, cancellation, and results.
- Operational limits based on Lambda/API/SQS/SES constraints verified during phase planning.

**Recommended first scope:**

1. Async resend for `email_failed` reports for one explicit week/status filter.
2. Preview plus operator reason.
3. Bounded target count and cooperative cancellation.
4. Per-target results and immutable audit.

**Defer initially:** Async generation retry across an incident if it requires long Bedrock/report generation workflows with uncertain runtime. Add after resend job mechanics and audit are proven.

### Immutable Audit Logs

**Operator expectation:** Every report recovery decision and mutation leaves read-only evidence showing who did what, when, why, which target was involved, and what happened.

**Complexity:** High.

**Core dependencies:**

- Append-only audit event write path.
- Audit event query access patterns by report target and by job ID.
- Admin-only audit read API/UI.
- Explicit field allowlist to avoid private artifacts and excessive PII.
- Tests that audit records are created for success, refused, failed, cancelled, and job lifecycle events.

**Recommended first scope:**

1. Audit event schema and writer.
2. Events for existing single retry, single resend, selected bulk resend.
3. Events for new async job lifecycle and per-target attempts.
4. Read-only audit timeline in report detail and job detail.

**Defer initially:** Full compliance archive/export retention policy beyond append-only operational evidence unless legal requirements are specified.

### Production Admin Browser Smoke

**Operator expectation:** Before support uses the UI, a real admin session proves production UI and API integration work in a browser and still preserve metadata-only boundaries.

**Complexity:** Medium.

**Core dependencies:**

- Approved source of real admin session/token that does not create temporary production admin accounts.
- Production frontend URL and API URL.
- Browser automation that can use the session without committing secrets.
- A safe read-only target/filter with known report operations rows, or graceful empty-state assertions.

**Recommended first scope:**

1. Load `https://app.stoaedu.ch/admin/report-operations` with an approved admin session.
2. Verify admin route renders and calls production API.
3. Apply a safe filter, open detail if a row exists, and verify action eligibility UI.
4. Assert no private artifact markers appear in DOM/network-visible responses.
5. Save screenshot/log artifact for release evidence.

**Defer initially:** Automated production mutation smoke from the browser. Keep mutation smoke as a separate, explicitly configured safe-fixture workflow.

### CI/CD Lambda Dist Guard

**Operator expectation:** CDK diff/deploy cannot accidentally package stale or missing `stoa-backend/dist` Lambda code.

**Complexity:** Medium.

**Core dependencies:**

- Canonical backend Lambda build command.
- Guard script or CI step that checks `dist` presence, expected modules, compile validity, source freshness, and fingerprint metadata.
- Infra CI integration before `cdk diff` and `cdk deploy`.
- Local runbook command for operators doing manual CDK work.

**Recommended first scope:**

1. Build/fingerprint `dist` from current backend source before infra diff/deploy.
2. Fail if `dist/stoa` is missing, if report operation modules are absent, or if fingerprint does not match current source commit.
3. Print clear remediation command.
4. Record fingerprint in CI logs and optionally deployment evidence.

**Defer initially:** Complex artifact registry or fully separate Lambda deployment pipeline unless current CDK asset workflow continues to cause drift after the guard.

## MVP Recommendation

Prioritize:

1. **Immutable audit foundation** - Build append-only audit events first so every new incident job action has durable evidence from day one.
2. **Async resend incident job** - Start with `email_failed` resend for one explicit week/status scope because it reuses the safest existing operation and avoids broad regeneration complexity.
3. **Read-only production admin browser smoke** - Close the v1.5 residual production UI risk without mutating customer data.
4. **Lambda dist guard** - Add the CI/CD and local preflight guard before any v1.6 deploy that could touch Lambda assets.

Defer:

- **Async incident-wide generation retry:** Higher risk because it can invoke generation/Bedrock/artifact rewrite paths across many targets. Add after async resend, cancellation, and audit are proven.
- **Support ticket integration:** Useful for workflow, but the required reason/incident reference field is enough for v1.6.
- **Report editing, PDF, multilingual, billing, analytics:** Product expansion, not operations hardening.
- **Default mutation browser smoke:** Keep production browser smoke read-only unless a named safe fixture and explicit approval path exist.

## Operator Safety And Privacy Boundaries

- All v1.6 surfaces remain admin-only.
- Incident jobs must require explicit scope, operation type, preflight preview, confirmation, and reason.
- No job should mutate a target unless current backend eligibility is true at execution time.
- Cancellation stops future attempts; it does not roll back already completed sends or generations.
- UI, API, audit logs, smoke artifacts, and CI logs must not expose `weekly-reports/`, `json_s3_key`, `html_s3_key`, `s3_key`, public S3 URLs, presigned URLs, or raw report HTML/JSON.
- Audit storage should use structured allowlisted metadata, not raw payload dumps.
- Production browser smoke should default to read-only inspection and privacy marker checks.
- CI/CD guard should fail before CDK diff/deploy if Lambda assets are missing or stale.
