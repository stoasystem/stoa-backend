# Plan 14-01: CDK and Runtime Reports Bucket Verification

**Phase:** 14 - CDK & Runtime Configuration Verification
**Status:** Planned
**Requirements:** INFRA-01, INFRA-02, INFRA-03, INFRA-04, INFRA-05

## Objective

Produce source/synth evidence that the reports bucket and Lambda runtime wiring are correct, and harden backend production behavior so missing `S3_REPORTS_BUCKET` cannot silently fall back to the local placeholder.

## Tasks

1. Add a backend reports bucket accessor in `src/stoa/config.py`.
   - Preserve local default behavior for `ENVIRONMENT != production`.
   - Raise `ValueError` in production when `s3_reports_bucket` is blank or exactly `stoa-reports`.
   - Return the trimmed configured bucket name otherwise.

2. Route report artifact S3 writes through the accessor.
   - Update `store_and_send_weekly_report` to use the validated bucket name.
   - Keep JSON and HTML write ordering unchanged.
   - Do not add S3 ACL parameters.

3. Add focused backend tests.
   - Assert local/development can still use the placeholder.
   - Assert production rejects placeholder and blank reports bucket values.
   - Assert artifact writes use the configured bucket when valid.

4. Capture CDK/source verification evidence.
   - Inspect `StorageStack` for private block-public-access, SSE-S3 encryption, access logging, and retain policy on `StoaReportsBucket`.
   - Inspect `ApiStack` for `S3_REPORTS_BUCKET` injection on API and weekly report Lambdas.
   - Inspect `ApiStack` for `reports_bucket.grant_read_write` on both Lambdas.
   - Run a CDK synth-style command if available, and record whether it produced usable evidence.

5. Write phase verification artifact.
   - Record exact commands and outcomes.
   - Separate source/synth confidence from deployed AWS confidence.
   - Mark AWS deployed env/IAM verification incomplete if not run.

- INFRA-03: Evidence records weekly report Lambda `S3_REPORTS_BUCKET` CDK env injection.
