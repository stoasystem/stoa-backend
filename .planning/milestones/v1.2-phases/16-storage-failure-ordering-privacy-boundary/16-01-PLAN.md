# Plan 16-01: Storage Failure Ordering and Privacy Boundary

**Phase:** 16 - Storage Failure Ordering & Privacy Boundary
**Status:** Planned
**Requirements:** STORAGE-05, STORAGE-06, STORAGE-07, PRIVACY-01, PRIVACY-02, PRIVACY-03

## Objective

Make the existing ordering and parent-access privacy guarantees explicit in tests and evidence.

## Tasks

1. Strengthen storage failure tests.
   - Assert a second S3 write failure occurs after exactly one successful artifact write.
   - Assert no `put_report`, status update, or SES send occurs after that partial S3 failure.
   - Keep success-order tests proving `s3`, `s3`, `put_report`, `ses`.

2. Strengthen parent report privacy tests.
   - Assert generated parent report detail responses omit `s3_key`, `html_s3_key`, `json_s3_key`, public URL, and presigned URL fields.
   - Keep existing ownership tests that reject unlinked child report reads before repository access.

3. Add a privacy audit note.
   - Record route/source checks showing report access is backend-mediated through parent routes.
   - Record that `/files/presign` remains image upload-only and does not expose report artifacts.
   - Record residual risk around orphaned first artifacts and deployed smoke.
