# Roadmap: v8.0 External Rollout Approval And Live Evidence Execution

**Status:** Completed
**Created:** 2026-07-09
**Prior milestone:** v7.4 Post-Launch Scale Customer Success And Growth Governance

## Goal

Start v8 by converting the completed v7 gated rollout contracts into real external rollout action or an explicit hold. v8.0 must gather final live approvals, current production evidence, scoped rollout accounts, support staffing, provider/mobile readiness, rollback controls, and user-visible limitations before any external rollout action.

## Reality Check

v7.0-v7.4 are complete as local gated operations contracts. They do not approve real controlled expansion, public launch, paid marketing, broad expansion, uncontrolled provider writes, or external customer rollout by themselves.

v8 is justified only if it stops being another contract chain. v8.0 must be the hard execution boundary: real approval and live evidence in, external rollout action or hold out.

## Function Purpose

- Convert v7 gate outputs into final live rollout approval or a current hold/remediation package.
- Verify production product paths with real or approved secret-backed sessions.
- Make external rollout action scoped, reversible, support-ready, and evidence-backed.
- Keep paid marketing and broad expansion separately gated.

## Implementation Strategy

- Use read-only production checks by default; allow only explicitly approved, scoped, reversible rollout actions.
- Record redacted evidence: timestamps, owners, request/build/deploy IDs, account aliases, rollout flags, blocker states, support coverage, and rollback authority.
- Treat missing provider/mobile/support/revenue/learning evidence as a hold blocker unless explicitly accepted or disabled for rollout.
- Close with external rollout start, controlled hold, rollback, or remediation.

## Phase Details

### Phase 447: Final External Rollout Approval And Scope Lock

**Goal**: Lock decision owner, rollout type, account/cohort scope, support coverage, disabled features, risk acceptance, and rollback authority.
**Depends on**: v7.4 Post-Launch Scale Customer Success And Growth Governance.
**Requirements**: V8ROLL-01
**Success Criteria**:

1. Final decision owner, product owner, support owner, teacher owner, finance owner, mobile owner, provider owners, incident owner, and release owner are recorded.
2. Rollout type, account/cohort scope, support coverage, disabled features, pricing posture, growth limits, risk acceptance, and rollback authority are locked.
3. Paid marketing and broad expansion approvals are tracked separately from external rollout approval.
4. Known limitations and disabled features are user/support visible.

### Phase 448: Live Product Provider Mobile Evidence Execution

**Goal**: Execute production evidence checks for account, payment, usage, notification, support, mobile, provider, learning, monitoring, and incident readiness.
**Depends on**: Phase 447.
**Requirements**: V8ROLL-02
**Success Criteria**:

1. Production checks cover login, verification, onboarding, entitlement, usage ledger, quota, checkout/paywall, notification, support, teacher operations, mobile/app-store, AI/provider, learning, monitoring, revenue, and incident readiness.
2. Evidence includes timestamp, owner, environment, request/build/deploy ID where applicable, blocker state, fallback, and rollback/disable control.
3. Any mutation is explicitly approved, scoped, reversible, and tied to rollout accounts.
4. Evidence excludes secrets, auth tokens, verification codes, raw provider payloads, raw student content, private object keys, and presigned URLs.

### Phase 449: Rollout Communications Support And User Limitation Readiness

**Goal**: Prepare user/support communications, known limitations, support macros, lifecycle messages, and escalation paths.
**Depends on**: Phase 448.
**Requirements**: V8ROLL-03
**Success Criteria**:

1. User communications, support macros, lifecycle messages, incident communications, disabled-feature copy, and known limitation copy are ready.
2. Support staffing, teacher coverage, finance/billing support, and escalation paths are active for rollout scope.
3. Preferences, locale, failed delivery, opt-out, refund, billing, login, mobile, AI/provider, and learning support paths are covered.
4. Communications distinguish launch, controlled rollout, hold, and disabled-feature states.

### Phase 450: External Rollout Smoke And Rollback Rehearsal

**Goal**: Run approved rollout smoke, dashboard checks, rollback rehearsal/tabletop, and launch-room handoff.
**Depends on**: Phase 449.
**Requirements**: V8ROLL-04
**Success Criteria**:

1. Approved smoke verifies rollout flags, auth, billing, usage, notification, support, mobile, learning, AI/help, monitoring, incident, and rollback paths.
2. Rollback rehearsal/tabletop or readback evidence is current.
3. Dashboards and launch-room ownership are active.
4. Start blockers are converted into owner/action rows with severity and next action.

### Phase 451: External Rollout Start Gate

**Goal**: Decide rollout start, hold, rollback, or remediation from current live evidence.
**Depends on**: Phase 450.
**Requirements**: VERIFY-89
**Success Criteria**:

1. Decision is rollout start, hold, rollback, or remediation.
2. Decision uses approval, production evidence, communications, support staffing, smoke, rollback, revenue, learning, mobile, provider, and incident readiness.
3. Roadmap, requirements, state, milestone snapshots, and project summary are updated.
4. v8.1 receives operating cadence, dashboards, owners, support coverage, and rollback controls only if rollout starts.
