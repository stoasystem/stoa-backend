# Roadmap: v5.9 Parent Admin Operations Visibility

**Status:** Planned
**Created:** 2026-07-03
**Depends on:** v5.8 Email Verification And Login Code Policy

## Goal

Give parents/customers and admins a complete support-grade view of entitlement, billing, usage, and verification state.

## Why This Is Its Own Milestone

Visibility is the final polish layer that makes the platform supportable. It should not be a few response fields hidden inside backend work; it needs dedicated API response design, frontend/admin handoff, support state explanations, tests, and release evidence.

## Scope

In scope:

- Parent/customer account state summary.
- Admin support account detail.
- Entitlement, billing, usage, and verification status aggregation.
- Support-safe explanations and timestamps.
- Frontend/admin handoff or backend-ready API contracts.
- Release gate for final core-ops closeout.

Out of scope:

- Full BI/warehouse dashboards.
- External support provider/CRM activation.
- Native app implementation.

## Planned Phases

- [ ] **Phase 217: Customer Account Operations Contract**
  - Define parent-facing plan, billing, entitlement, usage, and verification summary.
- [ ] **Phase 218: Admin Support Operations Contract**
  - Define admin account support detail, filters, timestamps, and bounded actions.
- [ ] **Phase 219: Backend Aggregation APIs**
  - Implement customer/admin aggregate response shapes.
- [ ] **Phase 220: Frontend/Admin Handoff And Focused Tests**
  - Update frontend handoff/docs or API fixtures; verify response shapes and role boundaries.
- [ ] **Phase 221: v5.9 Core Operations Closeout**
  - Close the final core-ops polish milestone and recommend the next product milestone.

## Phase Details

### Phase 217: Customer Account Operations Contract

**Goal**: Define what parents/customers can see about their own account state.
**Depends on**: v5.6 entitlement shape, v5.7 usage summary shape, and v5.8 verification state.
**Requirements**: OPSVIS-01
**Success Criteria**:

1. Response contract includes effective plan, billing status, entitlement source, current-period usage summary, verification state, and available actions.
2. Parent/customer fields avoid raw provider internals and internal support-only metadata.
3. Empty, free, paid, pending, past-due, canceled, and manually overridden account states are represented.
4. Frontend states are named for manage subscription, verify email, resend verification, quota blocked, and contact support.
5. Backward compatibility expectations for existing parent subscription endpoints are documented.

### Phase 218: Admin Support Operations Contract

**Goal**: Define the admin support view needed to answer account questions without raw database inspection.
**Depends on**: Phase 217.
**Requirements**: OPSVIS-02
**Success Criteria**:

1. Admin detail includes entitlement source, manual override state, billing provider state, recent provider event, usage summary/recent usage events, quota state, verification status, and support timestamps.
2. Admin filters and lookup paths cover parent, student, email, billing customer, and relevant status fields where feasible.
3. Support-safe explanations distinguish customer-facing reason from internal diagnostic reason.
4. Bounded admin actions are listed only when backed by existing safe primitives.
5. Role/admin-only boundaries are documented.

### Phase 219: Backend Aggregation APIs

**Goal**: Implement stable customer/admin aggregate responses by composing entitlement, billing, usage, and verification state.
**Depends on**: Phase 218.
**Requirements**: OPSVIS-03
**Success Criteria**:

1. Parent/customer account summary API composes entitlement, billing, usage, and verification data.
2. Admin account detail API composes the same domains with support-level detail.
3. Aggregation handles missing child, missing billing, missing ledger, and pending verification states without 500s.
4. Existing billing/admin endpoints remain backward compatible or have explicit migration notes.
5. Tests cover free, paid, pending, canceled, manually overridden, unverified, and no-usage accounts.

### Phase 220: Frontend/Admin Handoff And Focused Tests

**Goal**: Make the new account-state APIs ready for frontend/admin implementation.
**Depends on**: Phase 219.
**Requirements**: OPSVIS-04
**Success Criteria**:

1. Handoff document or fixtures show expected UI states for parent and admin.
2. API examples include successful, blocked, pending, unverified, and no-data states.
3. Role-boundary tests confirm parent-only and admin-only access.
4. Response-shape tests protect frontend contracts.
5. Known frontend implementation gaps are captured as future tasks if not implemented in this milestone.

### Phase 221: v5.9 Core Operations Closeout

**Goal**: Close the final core-operations polish milestone and choose the next product expansion.
**Depends on**: Phase 220.
**Requirements**: VERIFY-42
**Success Criteria**:

1. Parent/admin account-state contracts, backend aggregation, handoff, and tests are complete.
2. Final audit confirms v5.6-v5.9 entitlement, usage, verification, and visibility coverage.
3. Release evidence includes commit SHAs, test commands/output summaries, API examples, and known deferred items.
4. Roadmap, requirements, state, stoa_docs gap audit, and next milestone queue are updated.
5. Next product milestone recommendation is explicit: native app, rich curriculum editor frontend, production content import, live warehouse/BI, or external activation.

## Acceptance

- Parent/customer can see effective plan, billing status, usage summary, and verification state.
- Admin can inspect entitlement source, billing provider state, usage ledger summary, verification state, and support timestamps.
- Response shapes are frontend-ready.
- Tests cover key account states.
- Future native app can consume stable account-state APIs.
