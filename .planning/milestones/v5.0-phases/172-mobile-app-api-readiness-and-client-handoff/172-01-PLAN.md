# Plan 172-01: Define Mobile API Readiness And Client Handoff

## Goal

Create a client-ready mobile API inventory and handoff contract for core STOA flows.

## Work Items

1. Inspect current FastAPI route registration and route groups.
2. Inspect frontend auth/locale/notification/demo-fallback integration points.
3. Define route readiness groups for auth/session, student learning, parent reports, tutor tools, notifications, billing, support, curriculum, and admin status.
4. Define mobile client states for loading, empty, unavailable/error, unauthorized, stale/offline, refreshed, and no-demo-fallback behavior.
5. Map each route group to frontend/PWA reuse points and future native responsibilities.
6. Identify follow-up implementation targets for Phase 173, Phase 174, Phase 175, or future client workspaces.
7. Verify the handoff against Phase 172 success criteria.

## Verification

- Handoff covers all MOBILELOC-02 acceptance criteria.
- Route groups map to current backend route registrations.
- Frontend reuse and native follow-up responsibilities are explicit.
- No backend source changes are required unless a concrete route-contract gap is found.
