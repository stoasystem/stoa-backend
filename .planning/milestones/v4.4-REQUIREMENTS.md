# v4.4 Requirements: Live Payment Provider Rollout

**Status:** Active planning
**Started:** 2026-06-11
**Source:** `.planning/REQUIREMENTS.md`

## Goal

Move the v3.9 Stripe-first payment provider MVP toward controlled live rollout and operator-ready billing operations.

## Requirements

### PAYLIVE-01 Live Payment Rollout Contract And Credential Readiness

- Define Stripe live-mode credential path, webhook endpoint expectations, price/product mapping, environment variables, and rollback switches.
- Record TWINT production validation status and v4.4 scope boundary.
- Define safe smoke modes: local/test-mode verification, approved live-mode configuration inspection, and no-real-charge default.
- Map existing checkout/status/webhook code paths to implementation targets.

### PAYLIVE-02 Production Checkout And Webhook Verification

- Distinguish live-readiness configuration from test/local behavior.
- Record provider mode, event type, processing result, idempotency status, and request/correlation identifiers during webhook processing.
- Expose operator-visible checkout/webhook lifecycle status without exposing provider secrets.
- Cover live-readiness config, idempotency, failure states, and fallback behavior with focused tests.

### PAYLIVE-03 Refunds Invoices Tax And Dunning Readiness

- Define or implement refund readiness for eligible billing states and operator handoff.
- Surface invoice/receipt metadata or provider-hosted links where available.
- Define tax/accounting handoff metadata needed for Swiss accounting workflows.
- Define dunning state handling for overdue/payment-failed billing states and parent/admin visibility.

### VERIFY-27 v4.4 Payment Release Gate And Support Audit

- Run focused backend verification and relevant static checks.
- Update planning, feature-gap, and remaining-feature docs.
- Capture provider configuration, webhook, checkout, refund/invoice/dunning readiness evidence.
- Recommend the next milestone from the remaining feature queue.
