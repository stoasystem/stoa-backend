<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| admin request -> queue/list route | Query filters and page tokens must not bypass admin auth, bounds, or token scope validation. |
| delivery storage -> API response | Stored metadata must be reduced through an allowlist before reaching operators. |
| retry status -> operator action | Retry eligibility must not imply a mutation path that does not exist in Phase 150. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-150-01 | Elevation of Privilege | list/detail routes | mitigate | Keep both endpoints behind `require_role(\"admin\")`; tests assert repo helpers are not called for non-admin users. |
| T-150-02 | Information Disclosure | list/detail responses | mitigate | Use allowlisted response shaping and privacy tests over summaries, details, and audit metadata. |
| T-150-03 | Tampering | pagination tokens | mitigate | Use scoped encode/decode helpers and return 400 before repo calls for invalid tokens. |
| T-150-04 | Repudiation | recent activity ordering | mitigate | Add recent-first feed query rows and backfill/read-through for pre-feed delivery summaries instead of relying solely on unordered table scan semantics. |
| T-150-05 | Denial of Service | unbounded queue/audit queries | mitigate | Enforce FastAPI `limit`/`audit_limit` bounds and repository `Limit` values. |
| T-150-06 | Repudiation | retry ambiguity | mitigate | Expose read-only retry metadata and defer mutation until lifecycle transition semantics exist. |
</threat_model>
