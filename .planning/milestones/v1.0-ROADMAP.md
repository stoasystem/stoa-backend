## Overview

This milestone moved the parent portal from mock/demo-backed flows to authorized real backend data. It confirmed CDK, DynamoDB, Cognito, and configuration assumptions; delivered secure `/parents/me/...` backend routes; added real child summary/history/report data; aligned frontend services and pages without silent demo fallback; and completed focused backend/frontend verification.

Scope intentionally excluded automatic weekly report generation, EventBridge targets, SES workflows, PDF generation, and broad frontend redesign.

## Milestone Summary

### Key Decisions

- Use `/parents/me/...` as the normal parent portal contract.
- Use local DynamoDB parent profile `user_id` as the canonical parent ownership identifier.
- Keep existing path-ID parent endpoints compatible for legacy/admin use.
- Treat weekly report automation as a follow-up milestone.
- Keep report routes focused on stored DynamoDB report state, not scheduled generation or S3 artifacts.
- Remove silent demo fallback from parent-critical frontend flows.

### Issues Resolved

- Frontend and backend parent route contract mismatch.
- Parent portal dependency on demo fallback for child, summary, history, and report flows.
- Raw JWT `sub` versus local user ID ambiguity in parent ownership checks.
- Missing report states returning unstable or fabricated behavior.
- Lack of focused backend/frontend tests for parent ownership and empty/missing states.

### Issues Deferred

- Weekly report automation, Bedrock narrative generation, S3 artifacts, EventBridge target, SES email, monitoring, and retry behavior.
- Scalable indexed child lookup for high-volume parent-child relationships.
- S3 report artifact Lambda environment and permission wiring.
- Cleanup of unused parent practice-summary demo fallback path.

### Technical Debt Incurred

- Scan-based child lookup remains accepted for MVP scale.
- Phase 4 frontend edits live in `/Users/zhdeng/stoa-frontend`, so backend planning archives reference commits from a sibling repository.
