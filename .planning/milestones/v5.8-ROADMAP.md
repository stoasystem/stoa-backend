# Roadmap: v5.8 Email Verification And Login Code Policy

**Status:** Planned
**Created:** 2026-07-03
**Depends on:** v5.7 Usage Ledger And Quota Reconciliation

## Goal

Replace placeholder account verification behavior with explicit, customer-facing email verification and login-code policy.

## Why This Is Its Own Milestone

Verification touches account creation, login, support, messaging, Cognito compatibility, and frontend UX. It needs a complete state machine and careful compatibility plan rather than a quick endpoint patch.

## Scope

In scope:

- Email verification state machine.
- Verification code generation, storage, expiry, resend, attempt limit, and verification.
- Registration behavior update.
- Login-code policy decision and implementation or explicit product de-scoping.
- Forgot/reset password compatibility.
- Customer/admin verification status.

Out of scope:

- Replacing Cognito password auth without a token-compatible design.
- Broad identity-provider migration.
- SMS/WhatsApp verification.

## Planned Phases

- [ ] **Phase 212: Verification Policy And Cognito Compatibility Contract**
  - Decide password-only plus email verification vs login-code flow; document token/session implications.
- [ ] **Phase 213: Email Verification Code Service**
  - Add code lifecycle, expiry, resend, attempt limits, and storage.
- [ ] **Phase 214: Registration And Account-State Integration**
  - Replace `admin_marked_verified` default with real state unless internal bypass is enabled.
- [ ] **Phase 215: Login-Code Policy Implementation And UI/API Handoff**
  - Implement supported login-code behavior or remove unsupported API/UI expectations.
- [ ] **Phase 216: v5.8 Verification Release Gate**
  - Verify success, wrong code, expired code, resend, already verified, forgot/reset compatibility, docs, and v5.9 handoff.

## Phase Details

### Phase 212: Verification Policy And Cognito Compatibility Contract

**Goal**: Decide the supported verification and login-code behavior before implementation touches auth flows.
**Depends on**: current Cognito auth/register/forgot/reset behavior from Phase 201.
**Requirements**: VERIFYACC-01
**Success Criteria**:

1. Email verification states are defined, including pending, sent, verified, expired, failed, resent, and internal bypass.
2. Login-code policy is explicit: password-primary plus email verification, or a token-compatible login-code/custom-auth design.
3. Cognito session/token implications are documented so the backend does not create a fake login flow.
4. Registration, login, refresh, logout, forgot, and reset compatibility rules are documented.
5. Frontend/API handoff lists the supported user-facing states and actions.

### Phase 213: Email Verification Code Service

**Goal**: Implement the backend lifecycle for verification codes.
**Depends on**: Phase 212.
**Requirements**: VERIFYACC-02
**Success Criteria**:

1. Code generation, safe storage, expiry, attempt count, resend count, and last-sent timestamp are implemented.
2. Resend invalidates or supersedes the previous active code according to the contract.
3. Wrong code, expired code, too many attempts, already verified, and success paths are deterministic.
4. Email sending can use the approved existing messaging abstraction or a documented internal-development fallback.
5. Tests cover code validation and lifecycle edge cases.

### Phase 214: Registration And Account-State Integration

**Goal**: Make new-account verification state truthful across profile and auth outputs.
**Depends on**: Phase 213.
**Requirements**: VERIFYACC-03
**Success Criteria**:

1. Registration no longer presents normal users as product-verified by default.
2. Internal/admin bypass, if kept for development, is explicit and visible in account state.
3. Auth profile output exposes verification status without breaking existing clients.
4. Verification success updates the persisted user/account state.
5. Forgot/reset password remains compatible with the new verification state.

### Phase 215: Login-Code Policy Implementation And UI/API Handoff

**Goal**: Implement the chosen login-code policy or remove unsupported expectations.
**Depends on**: Phase 214.
**Requirements**: VERIFYACC-04
**Success Criteria**:

1. If password login remains primary, API/docs explicitly distinguish email verification codes from login codes.
2. If login-by-code is required, the implementation produces valid Cognito-compatible sessions/tokens.
3. Frontend handoff includes request, resend, confirm, error, expired, and already-verified states.
4. Unsupported UI/API expectations are removed or clearly marked out of scope.
5. Tests cover supported policy behavior and protect existing password auth.

### Phase 216: v5.8 Verification Release Gate

**Goal**: Close v5.8 as a complete account verification milestone.
**Depends on**: Phase 215.
**Requirements**: VERIFY-41
**Success Criteria**:

1. Verification contract, code service, registration integration, login-code policy, and handoff are complete.
2. Focused backend tests pass for success, wrong code, expired code, resend, already verified, and forgot/reset compatibility.
3. Documentation records the chosen policy and any intentionally deferred identity-provider work.
4. Release evidence includes commit SHA, test command/output summary, and known deferred items.
5. v5.9 operations visibility handoff is updated with verification fields.

## Acceptance

- New accounts have a truthful verification status.
- Verification codes expire and enforce attempt limits.
- Resend and already-verified behavior are deterministic.
- Login-code policy is product-true and token-compatible.
- Customer/admin views can show verification state.
