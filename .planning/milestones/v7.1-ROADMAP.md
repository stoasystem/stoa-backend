# Roadmap: v7.1 Controlled Expansion Operations And Remediation

**Status:** Completed
**Created:** 2026-07-09
**Prior milestone:** v7.0 Final Live Approval And Controlled Expansion Start

## Goal

Operate the approved controlled expansion cohort and fix the real bottlenecks that appear under larger but still bounded usage. v7.1 must turn expansion evidence into shipped remediation before any public launch preparation.

## Function Purpose

- Validate whether STOA can support more families without losing support quality, learning quality, revenue correctness, or operational control.
- Fix high-impact issues across account, billing, usage, support, teacher operations, mobile, AI/provider, curriculum, notifications, and reliability.
- Preserve rollback and cohort pause authority.

## Implementation Strategy

- Run daily controlled-expansion review with product, support, teacher, finance, mobile, provider, and incident owners.
- Rank issues by severity, frequency, support load, revenue impact, learning impact, and rollout risk.
- Ship focused fixes with regression tests, release evidence, support copy, and rollback notes.
- Close with public-launch-prep, continue expansion, hold, rollback, or another remediation cycle.

## Phase Details

### Phase 427: Controlled Expansion Day-One Operations

**Goal**: Operate cohort enablement, support coverage, dashboards, incident checks, and rollback readiness.
**Depends on**: v7.0 Final Live Approval And Controlled Expansion Start.
**Requirements**: V7EXPAND-01
**Success Criteria**:

1. Cohort enablement, support coverage, teacher coverage, dashboards, incident checks, pause criteria, and rollback controls are active.
2. Daily evidence distinguishes real users from internal, fixture, dry-run, and test traffic.
3. Support, teacher, finance, mobile, provider, and incident owners review signals.
4. Any severe issue triggers pause, rollback, or hotfix review.

### Phase 428: Expansion Account Revenue Support Fixes

**Goal**: Fix account, verification, entitlement, usage/quota, billing, lifecycle, support, and parent clarity issues.
**Depends on**: Phase 427.
**Requirements**: V7EXPAND-02
**Success Criteria**:

1. Account, verification, entitlement, subscription, usage/quota, billing, lifecycle, support, and parent clarity issues are fixed or explicitly deferred.
2. Revenue-impacting fixes are auditable, reversible, and support-visible.
3. Support macros and parent copy are updated for changed behavior.
4. Regression coverage exists for high-impact fixes.

### Phase 429: Expansion Learning Mobile Teacher Provider Fixes

**Goal**: Fix learning, AI/provider, teacher dispatch, mobile, notification, curriculum, and parent progress issues.
**Depends on**: Phase 428.
**Requirements**: V7EXPAND-03
**Success Criteria**:

1. Learning, AI/provider, teacher dispatch, mobile, notification, curriculum, recommendation, and parent progress issues are fixed or explicitly deferred.
2. Curriculum changes remain specially authorized.
3. AI behavior remains reviewed or policy-bound.
4. Mobile/provider fallbacks are visible to support and users when needed.

### Phase 430: Expansion Reliability Release Evidence

**Goal**: Add regression tests, smoke evidence, release notes, rollback notes, and operator runbook updates.
**Depends on**: Phase 429.
**Requirements**: V7EXPAND-04
**Success Criteria**:

1. Fixes include tests, smoke evidence, release notes, rollback notes, and operator runbook updates.
2. Release evidence links code SHA, deploy/build IDs, request IDs where applicable, timestamp, and owner.
3. Dashboards and alerts distinguish test, controlled expansion, and real customer traffic.
4. Known residual risks are owner-assigned.

### Phase 431: Expansion Outcome Gate

**Goal**: Decide public-launch-prep, continue expansion, hold, rollback, or remediation.
**Depends on**: Phase 430.
**Requirements**: VERIFY-85
**Success Criteria**:

1. Decision is public-launch-prep, continue expansion, hold, rollback, or remediation.
2. Decision uses activation, support, teacher, revenue, learning, mobile, provider, incident, and release evidence.
3. Roadmap, requirements, state, milestone snapshots, and project summary are updated.
4. v7.2 is allowed only if expansion evidence supports launch preparation.
