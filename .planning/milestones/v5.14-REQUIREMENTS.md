# Requirements: v5.14 Verification And Login Reliability

**Milestone:** v5.14
**Status:** Planned
**Created:** 2026-07-05

## Requirements

### AUTHREL-01 Verification Reality Audit

- Map current registration, Cognito confirmation, local verification metadata, resend, confirm, login, and frontend state behavior.
- Identify implemented, stubbed, provider-dependent, and broken paths.
- Document one canonical account-entry policy before implementation.

### AUTHREL-02 Email Verification Completion

- Resend and confirm endpoints are idempotent, bounded, and frontend-compatible.
- Verification state is reflected consistently in login, profile, parent/admin support, and account operations surfaces.
- Failure states are actionable and do not leak provider internals.

### AUTHREL-03 Login-Code Policy

- If login-code/passwordless is supported, backend and frontend both implement the full lifecycle.
- If not supported yet, UI/API references are disabled or return a clear unsupported-policy response.
- Tests prove no half-enabled login path remains.

### AUTHREL-04 Account Activation Support State

- Admin/support can inspect bounded activation, verification, resend, and provider-delivery metadata.
- Teacher/admin pending-review or activation state is represented clearly.
- Recovery paths are documented and tested.

### VERIFY-48 Verification And Login Reliability Gate

- Focused backend/frontend tests pass for verification and login reliability.
- Delivery/provider failures are represented in test fixtures.
- Docs and release evidence record remaining provider or rollout dependencies.
