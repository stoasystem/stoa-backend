# Requirements: v5.13 Payment And Entitlement Production Completion

**Milestone:** v5.13
**Status:** Planned
**Created:** 2026-07-05

## Requirements

### PAYPROD-01 Payment Reality Audit

- Map current backend checkout, subscription, entitlement, usage-limit, admin support, and frontend paywall behavior.
- Identify which flows are implemented, stubbed, locally verified, or blocked by provider credentials.
- Produce a short evidence table before implementation starts.

### PAYPROD-02 Checkout And User Paid-State Flow

- Parent-facing checkout/paywall state uses real backend API state, not demo fallback.
- Pending, active, failed, canceled, refunded, and manual-override states render clearly.
- Backend returns enough status for frontend access decisions and support handoff.

### PAYPROD-03 Webhook Reconciliation And Entitlement Activation

- Provider events are idempotently stored/reconciled.
- Successful payment activates entitlement exactly once.
- Duplicate, stale, missing, or conflicting events produce support-visible reconciliation status.
- Entitlement and usage-limit checks agree after reconciliation.

### PAYPROD-04 Billing Support Evidence

- Admin/support view exposes bounded payment, entitlement, invoice, refund, and reconciliation metadata.
- Raw provider secrets, full payloads, and sensitive payment details are not exposed.
- Manual override state is visible and distinguishable from provider-backed entitlement.

### VERIFY-47 Payment Production Completion Gate

- Focused backend tests, frontend tests, and release evidence pass for paid access.
- Live smoke is documented as blocked or completed depending on provider credential availability.
- Remaining gaps are promoted to future requirements rather than hidden in completion notes.
