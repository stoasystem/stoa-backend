# Roadmap: v5.35 Real Pilot Blocker Burn-Down And Launch Execution

**Status:** Complete
**Created:** 2026-07-07
**Prior milestone:** v5.30-v5.34 Live Execution Contracts

## Goal

Convert the current metadata-only launch contracts into an executable internal pilot by clearing or explicitly disabling the remaining start blockers, wiring the operational evidence into the existing gates, and starting the first real cohort only when the live gate returns `start_limited_pilot`.

## Reality Check

v5.30-v5.34 are contract-complete, not real-world complete. They added support-safe evidence surfaces for pilot approval, live pilot execution, remediation, controlled expansion, and public launch operations, but they do not prove that real pilot users, provider writes, expansion, paid marketing, or public launch have occurred.

The immediate product problem is therefore not another abstract launch phase. The next milestone must burn down the exact blockers that keep the live gate in `hold`.

## Function Purpose

- Turn payment, notification, support CRM, BI/APM, mobile build, restore/tabletop, and launch-room gaps into real approval evidence or deliberate pilot disablement.
- Create the first narrow cohort start package with owners, rollback authority, support coverage, and user-facing disabled-feature copy.
- Execute the existing live gate against current evidence and record a start, hold, or harden decision.
- Prevent accidental public launch, paid marketing, or uncontrolled provider writes.

## Implementation Strategy

- Use the existing `production_pilot_service.live_pilot_safe_start_gate_execution` and related evidence contracts as the source of truth.
- Prefer real provider activation where available; when not available, record explicit pilot disablement with owner, user impact, and fallback path.
- Keep pilot scope narrow: known parent/student accounts, approved teacher/support owners, staged flags, and immediate rollback.
- Close with an operator-ready launch packet and a v5.36 handoff only if the decision is `start_limited_pilot`.

## Phase Details

### Phase 347: Live Blocker Inventory And Owner Assignment

- `production_pilot_service.real_pilot_blocker_inventory_owner_assignment`
- Converts current live-gate blockers into an owner/action table.
- Defaults to blocked until owner approvals, provider/mobile evidence, and operations evidence are complete.

### Phase 348: Provider Or Disablement Activation Closeout

- `production_pilot_service.provider_or_disablement_activation_closeout`
- Records each provider/mobile dependency as verified, disabled, read-only, or open.
- Keeps participant copy, fallback path, and rollback controls support-safe.

### Phase 349: Pilot Cohort Account And Support Dry Run

- `production_pilot_service.pilot_cohort_account_support_dry_run`
- Validates account and support readiness without enabling customer mutation.
- Requires start gate, approved accounts, and support staffing before dry-run readiness is marked ready.

### Phase 350: Launch Room Restore And Incident Readiness Closeout

- `production_pilot_service.launch_room_restore_incident_readiness_closeout`
- Combines restore/tabletop evidence with dashboard, alert routing, pause, rollback, and provider-disablement readiness.

### Phase 351: Real Pilot Start Decision Gate

- `production_pilot_service.real_pilot_start_decision_gate`
- Returns `start_limited_pilot` only when blocker inventory, provider closeout, dry run, and launch-room readiness have no blockers.
- v5.36 real operations remain blocked by default.
