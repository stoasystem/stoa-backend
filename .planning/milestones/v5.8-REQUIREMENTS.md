# Requirements: v5.8 Email Verification And Login Code Policy

**Milestone:** v5.8
**Status:** Planned
**Created:** 2026-07-03

## Requirements

### VERIFYACC-01 Verification Policy Contract

Acceptance criteria:

- Supported policy is explicit: password-primary plus email verification, or token-compatible login-code/custom-auth.
- Email verification states include pending, sent, verified, expired, failed, resent, and internal bypass if allowed.
- Cognito session/token compatibility is documented.
- Registration, login, refresh, logout, forgot, and reset interactions are documented.
- Frontend/API handoff states and user actions are listed.

### VERIFYACC-02 Email Verification Code Service

Acceptance criteria:

- Service can request, generate, validate, resend, expire, and mark verification success/failure.
- Stored code material avoids plain text where practical and includes expiry, attempts, resend count, and last-sent timestamp.
- Resend behavior invalidates or supersedes previous active codes.
- Wrong, expired, too-many-attempts, already-verified, and success outcomes are deterministic.
- Tests cover code lifecycle edge cases.

### VERIFYACC-03 Registration And Account State Integration

Acceptance criteria:

- Normal registration no longer presents accounts as product-verified by default.
- Internal/admin bypass, if retained, is explicit in persisted state.
- Auth/account profile output exposes verification status without breaking existing clients.
- Verification success updates the persisted user/account state.
- Forgot/reset password behavior remains compatible.

### VERIFYACC-04 Login-Code Policy Implementation

Acceptance criteria:

- If password login remains primary, email verification codes are not described as login codes.
- If login-by-code is implemented, it produces valid session/token behavior.
- Frontend/API handoff covers request, resend, confirm, expired, wrong-code, and already-verified states.
- Unsupported UI/API expectations are removed or documented out of scope.
- Tests protect existing Cognito password login, refresh, logout, forgot, and reset flows.

### VERIFY-41 v5.8 Verification Release Gate

Acceptance criteria:

- Verification contract, code service, registration integration, login-code policy, and handoff are complete.
- Focused backend tests pass.
- Release evidence records commit SHA, test command/output summary, and deferred identity-provider work.
- Customer/admin verification visibility handoff is ready.
- v5.9 operations visibility handoff is updated.
