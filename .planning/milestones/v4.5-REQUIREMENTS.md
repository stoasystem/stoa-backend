# v4.5 Requirements: Support Evidence Integrations And Operations Handoff

**Status:** Active planning
**Created:** 2026-06-12
**Source:** `.planning/NEXT-MILESTONES.md`, `.planning/research/SUMMARY.md`

## Goal

Connect STOA's existing support-safe evidence packages and operations metadata to approved support destinations, reducing manual copy/download handoff while preserving metadata-only evidence boundaries.

The milestone should prove a controlled destination workflow, not a broad CRM platform. Existing `preview`, `copy`, and `download` modes remain supported, and unapproved external writes remain fail-closed.

## Requirements

### SUPPORTINT-01 Support Destination Contract And Credential Readiness

- Define approved destination modes, including manual modes plus candidate `internal_queue`, `shared_mailbox`, `zendesk_ticket`, `freshdesk_ticket`, and `helpscout_conversation`.
- Identify credential references, environment variables, secret ownership, provider account requirements, and operator approval gates for each selected destination.
- Expose configured, missing, refused, and dry-run-safe readiness states without exposing secrets.
- Define metadata-only payload rules, allowed provider fields, attachment policy, redaction rules, and outbound payload digest behavior.
- Preserve existing `external_write` refusal unless replaced by an explicitly approved destination mode with tests.

### SUPPORTINT-02 Support Evidence Export Destination Integration

- Add backend delivery for support-safe evidence packages to one approved destination path.
- Validate destination readiness and package privacy before provider calls.
- Keep existing preview/copy/download behavior available as fallback.
- Record lifecycle status, request IDs, timestamps, idempotency key, provider object references, retry count, and redacted refusal/failure reasons.
- Avoid exposing private report artifacts, raw payloads, auth tokens, authorization headers, presigned URLs, or customer-sensitive provider data.

### SUPPORTINT-03 Operator Queue And Handoff Status Visibility

- Expose admin-only support handoff status for created, queued, sent, failed, refused, and retried packages.
- Add bounded filters and detail views for recent support handoff activity.
- Include provider references, redacted failure/refusal reasons, retry count, correlation ID, and privacy validation summary.
- Keep retry behavior explicit, bounded, and unavailable for privacy-failed or unapproved destinations.
- Keep support workflow usable without requiring broad CRM automation.

### VERIFY-28 v4.5 Support Integration Release Gate

- Run focused backend and frontend checks for the selected support integration path, status visibility, and manual fallback.
- Capture destination credential/configuration status with secrets redacted.
- Verify refusal behavior for unapproved destinations, missing credentials, provider failures, duplicate retries, and privacy violations.
- Update roadmap, state, feature-gap audit, and remaining-feature queue after v4.5.

## Future Requirements

- Broad CRM automation and customer messaging workflows.
- Two-way ticket synchronization.
- Support SLA analytics beyond first-pass handoff status.
- Live support destination writes for additional vendors.
- Native mobile support flows.

## Out of Scope

- Unapproved external writes.
- Raw report artifact exposure.
- Direct S3 artifact links or presigned URLs.
- Storing provider credentials in code, planning docs, payloads, or audit rows.
- Broad customer messaging campaigns.
- Replacing existing metadata-only report operations evidence boundaries.
