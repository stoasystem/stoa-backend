# Requirements: v5.35 Real Pilot Blocker Burn-Down And Launch Execution

**Milestone:** v5.35
**Status:** Planned
**Created:** 2026-07-07
**Prior milestone:** v5.30-v5.34 Live Execution Contracts

## Purpose

Move from launch contracts to a real internal pilot start decision. v5.35 must clear or explicitly disable the remaining blockers that keep the live pilot gate in `hold`, then run the gate with current evidence.

## Requirements

### PILOTSTART-01 Live Blocker Inventory And Owner Assignment

Acceptance criteria:

- Current blockers are reconciled from v5.30-v5.34 evidence, production configuration, provider state, mobile release state, and support operations.
- Each blocker has owner, action, due date, status, severity, user impact, and whether it blocks pilot start.
- Duplicate or stale blockers are removed with explanation.
- The blocker inventory distinguishes local contract gaps from real operational gaps.

### PILOTSTART-02 Provider Or Disablement Activation Closeout

Acceptance criteria:

- Payment, notification, support CRM, BI/APM, mobile/TestFlight, and other pilot dependencies are either activated for pilot scope or explicitly disabled.
- Activated providers have redacted evidence with timestamp, owner, environment, request/build ID where applicable, and rollback/disable control.
- Disabled dependencies include user-facing copy, support macro, fallback path, and owner approval.
- No secrets, raw provider payloads, raw student content, private object keys, or presigned URLs are recorded.

### PILOTSTART-03 Pilot Cohort Account And Support Dry Run

Acceptance criteria:

- Approved real admin, parent, student, and teacher/support accounts can complete the pilot-critical login, onboarding, learning, payment/entitlement, notification, and support paths in dry run.
- The dry run records redacted request IDs, account IDs, timestamps, and observed failures.
- Pilot cohort size, account list ownership, communication plan, consent state, and rollback authority are documented.
- Support has a clear escalation path for billing, login, learning quality, mobile, AI, and incident issues.

### PILOTSTART-04 Launch Room Restore And Incident Readiness Closeout

Acceptance criteria:

- Restore/readback or approved tabletop evidence is current for pilot-critical data.
- Launch-room staffing, incident severity, pause criteria, rollback criteria, and provider-disablement controls are documented.
- Monitoring covers auth, entitlement, usage ledger, notifications, support SLA, AI/provider health, mobile, and revenue-relevant flows.
- Any unresolved readiness gap is explicitly accepted, disabled for pilot, or marked start-blocking.

### VERIFY-69 Real Pilot Start Decision Gate

Acceptance criteria:

- The live safe-start gate is executed with current evidence.
- Decision is `start_limited_pilot`, `hold`, or `harden_further`.
- If the decision is `start_limited_pilot`, v5.36 receives cohort scope, owners, dashboards, rollback controls, and first-day operating plan.
- If the decision is `hold` or `harden_further`, v5.36 is not allowed to operate real users and the blocker list is the next execution target.

## Out of Scope

- Broad public launch.
- Paid marketing.
- Expansion beyond the approved cohort.
- Uncontrolled provider writes.
- New AI autonomy beyond reviewed/approved workflows.
