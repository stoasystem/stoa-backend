## Phase Decision Map

- D-01: Use local DynamoDB parent profile `user_id` as the canonical parent ownership identifier.
- D-02: Resolve authenticated parent identity with `user_repo.get_user(user["sub"])`, then Cognito email, then `user_repo.get_user_by_email(email)`.
- D-03: Require the resolved profile role to be `parent`; do not admit admin, student, teacher, or tutor through `/parents/me/...`.
- D-04: Do not trust client-supplied parent IDs in the normal parent portal flow.
- D-05: Implement child lookup by scanning student profile items where `role == "student"` and `parent_id == resolved_parent_user_id`.
- D-06: Paginate the scan across all DynamoDB pages.
- D-07: Return `{ "items": [...] }`, including `{ "items": [] }` for no linked children.
- D-08: Child list items include `id`, `userId`, `name`, `email`, `grade`, `subjects`, and `relationship`.
- D-09: Keep legacy `/parents/{parent_id}/children` and `/parents/{parent_id}/reports/{week}` available.
- D-10: If legacy parent routes are touched, compare requested `parent_id` with the resolved local parent profile ID rather than raw JWT `sub`.
- D-11: Keep admin compatibility for legacy path-ID routes only if it already exists; do not allow admin through `/parents/me/...`.
- D-12: Keep helper functions route-local in `src/stoa/routers/parents.py` unless a small repository helper clearly reduces duplication.
- D-13: Use existing FastAPI/Pydantic patterns from the current router.
- D-14: Add minimal tests for confidence; broader test/data coverage remains Phase 5.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cognito JWT to legacy path route | Authenticated role and identity claims determine whether a path-supplied parent ID can be used. |
| Legacy path route to DynamoDB/report repository | Route code reads child or report records for the requested `parent_id`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02-S | Spoofing | Legacy parent routes | mitigate | Parent users must resolve to a local profile and match path `parent_id` against `resolved.parent_user_id`. |
| T-02-02-T | Tampering | Path `parent_id` | mitigate | Client-supplied path IDs are accepted only after role and local-profile ownership checks for parent users. |
| T-02-02-I | Information disclosure | `list_children` and `get_report` | mitigate | Non-parent/admin roles are rejected; parents cannot request another parent's path ID; admin support remains explicit on path-ID routes. |
| T-02-02-E | Elevation of privilege | Admin compatibility | mitigate | Admin compatibility is retained only for legacy path routes and tests assert admin is denied on `/parents/me/children`. |
| T-02-02-R | Repudiation | Legacy route behavior | accept | Existing router does not add audit logging; this plan preserves behavior and verifies authorization boundaries. |
