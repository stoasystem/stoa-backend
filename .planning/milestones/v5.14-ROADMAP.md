# Roadmap: v5.14 Verification And Login Reliability

**Status:** Planned
**Created:** 2026-07-05
**Prior milestone:** v5.13 Payment And Entitlement Production Completion

## Goal

Make account verification and login-code behavior dependable across registration, resend, confirmation, login, account activation, and support recovery.

## Why This Is A New Milestone

Email verification, login code, and account activation are not polish. They decide whether real users can enter the product. Prior planning touched verification policy, but the next work should verify the current app reality and close the full user-facing reliability loop.

## Product Purpose

- New students, parents, teachers, and admins can complete account activation reliably.
- Verification resend and confirm flows are understandable and rate-limited.
- Login-code/passwordless behavior is either fully implemented or explicitly disabled with correct UI/API behavior.
- Support can see bounded account activation state without needing raw provider internals.
- Delivery failures are visible enough to debug during internal development.

## Implementation Strategy

- Audit current Cognito, local user profile, email verification, login-code, and frontend behavior before changing policy.
- Prefer one clear login path over half-enabled alternatives.
- Keep resend/confirm logic idempotent and bounded.
- Add rate limits and abuse controls where they protect real flows without slowing internal development.
- Add support-visible delivery and verification state.
- Close with focused tests and smoke fixtures for expired, wrong, duplicate, and already-confirmed codes.

## Workstreams

1. Account verification reality audit.
2. Email verification resend/confirm completion.
3. Login-code/passwordless policy implementation or explicit removal from UI.
4. Account activation and support visibility.
5. Reliability, abuse-control, and release gate.

## Acceptance

- Registration creates a profile and verification state that frontend can resolve.
- Resend/confirm works for valid users and fails clearly for invalid, expired, wrong, or already-used states.
- Login behavior is consistent with verification status and account activation status.
- Login-code/passwordless behavior is not exposed unless backend support is complete.
- Admin/support can inspect activation state and recent bounded verification events.
- Tests cover student, parent, teacher/admin activation, resend limits, wrong code, expired code, already confirmed, blocked account, and provider failure.

## Out Of Scope

- Native push login.
- New identity provider migration.
- Broad anti-fraud platform work beyond account-entry safeguards.
