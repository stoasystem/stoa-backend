# Requirements: v5.25 Pilot Activation Blocker Burn-Down And Safe Start Decision

**Milestone:** v5.25
**Status:** Active
**Created:** 2026-07-07
**Prior milestone:** v5.24 Limited Production Pilot And Launch Readiness

## Purpose

Convert the conditional v5.24 go/no-go decision into a real safe-start decision. v5.25 must clear or explicitly disable every required activation blocker before any limited pilot cohort is enabled.

## Requirements

### PILOTSTART-01 Pilot Activation Blocker Reality Audit

Acceptance criteria:

- Current readiness is re-checked across payment, Cognito/email, notifications, support CRM, BI/APM, mobile release/TestFlight, production restore, tabletop, staffing, cohort, data lifecycle, and rollback.
- Each blocker has owner, required action, pilot impact, fallback/disable option, evidence source, and decision deadline.
- Dependencies are classified as required for pilot, explicitly disabled for pilot, optional, blocked, or cleared.
- No real-user pilot start is recommended while required blockers remain uncleared.

### PILOTSTART-02 Provider Activation Or Explicit Disablement

Acceptance criteria:

- Payment, notification, support CRM, BI/APM, mobile distribution, and restore/tabletop dependencies either have approved live/read-only evidence or are explicitly disabled from pilot scope.
- Disabled dependencies have parent/student/support copy and operator impact documented.
- Approved live checks record redacted request IDs, timestamps, owners, and rollback controls.
- Provider evidence excludes secrets, tokens, raw provider payloads, raw student content, and private object keys.

### PILOTSTART-03 Pilot Environment Cohort And Account Dry Run

Acceptance criteria:

- Pilot cohort setup validates parent/student accounts, email verification, entitlement, child binding, curriculum placement, mobile install path, notification preference, and support contact path.
- Dry-run accounts prove onboarding, account recovery, support escalation, rollback communication, and exit path without customer impact.
- Fixture cleanup or retention is documented.
- Known unsupported pilot features are visible before pilot start.

### PILOTSTART-04 Launch Room Rehearsal And Safe Start Package

Acceptance criteria:

- Launch-room runbook defines owner, schedule, dashboard, alert routing, support staffing, escalation, rollback, and daily reporting.
- Rehearsal covers auth failure, payment blocker, notification blocker, AI/provider issue, support spike, mobile crash, and rollback.
- Safe-start package includes cohort list, enablement checklist, disabled-feature list, support plan, monitoring plan, rollback plan, and communications plan.
- Rehearsal evidence records timestamp, participants, findings, and unresolved blockers.

### VERIFY-59 Pilot Safe Start Gate

Acceptance criteria:

- Decision is one of: start limited pilot, hold, or harden further.
- Decision evidence includes blocker table, provider evidence, dry-run evidence, launch-room rehearsal, accepted risks, and next milestone recommendation.
- Roadmap, requirements, state, milestone snapshots, and project summary are updated.
- v5.26 scope is adjusted based on the actual start/hold decision.

## Out of Scope

- Broad public launch.
- Paid marketing or acquisition campaigns.
- Expanding cohort size before safe-start blockers are resolved.
- Hiding provider-disabled features from pilot participants.
- Production customer mutation outside approved pilot cohort and rollout controls.
