# Roadmap: v5.7 Usage Ledger And Quota Reconciliation

**Status:** Planned
**Created:** 2026-07-03
**Depends on:** v5.6 Effective Entitlements And Paid Access Enforcement

## Goal

Turn plan-governed usage from short-lived counters into a durable, queryable, reconcilable product ledger.

## Why This Is Its Own Milestone

Usage tracking is not just an implementation detail. It underpins quota enforcement, customer support, parent visibility, billing explanations, future analytics, and abuse/debug investigations. It deserves its own build/test/release cycle.

## Scope

In scope:

- Usage event model and repository/service.
- Ledger writes for question submit, OCR/AI answer generation, teacher-help request, chat, and hints.
- Counter compatibility and reconciliation policy.
- Admin usage query API.
- Parent/customer usage summary handoff.
- Focused tests and release evidence.

Out of scope:

- Warehouse/BI deployment.
- Full analytics dashboards.
- Payment provider activation.

## Planned Phases

- [ ] **Phase 207: Usage Ledger Contract And Access Patterns**
  - Define event schema, idempotency model, partition keys, query dimensions, retention expectations, and counter compatibility.
- [ ] **Phase 208: Usage Event Repository And Service**
  - Add repository/service for durable usage events and bounded summaries.
- [ ] **Phase 209: Plan-Governed Action Instrumentation**
  - Write usage events for question submit, OCR/AI answer, teacher-help request, chat, and hint flows.
- [ ] **Phase 210: Quota Reconciliation And Admin Usage Query**
  - Keep counters stable while exposing ledger/counter reconciliation and admin queries.
- [ ] **Phase 211: v5.7 Usage Ledger Release Gate**
  - Verify idempotency, quota behavior, query response shapes, docs, and v5.8 handoff.

## Phase Details

### Phase 207: Usage Ledger Contract And Access Patterns

**Goal**: Define the ledger model before adding writes to production paths.
**Depends on**: v5.6 entitlement source shape.
**Requirements**: USAGE-01
**Success Criteria**:

1. Event schema defines actor, subject/student, parent, product area, action, quantity, entitlement source, period, idempotency key, timestamp, and display-safe metadata.
2. Access patterns are listed for admin lookup, parent/customer summaries, reconciliation, and support investigation.
3. Partition/sort key strategy fits current DynamoDB single-table usage unless a justified new access path is documented.
4. Retention expectations and privacy boundaries are explicit.
5. Counter compatibility and migration risk are documented before instrumentation.

### Phase 208: Usage Event Repository And Service

**Goal**: Add reusable usage event persistence and summary reads.
**Depends on**: Phase 207.
**Requirements**: USAGE-02
**Success Criteria**:

1. Repository writes usage events idempotently by event ID or idempotency key.
2. Service exposes internal methods for recording and querying bounded usage events.
3. Summary helpers can aggregate by student, parent, period, product area, and entitlement source.
4. Duplicate write attempts do not create duplicate ledger rows.
5. Unit tests cover write, duplicate, query, and summary behavior.

### Phase 209: Plan-Governed Action Instrumentation

**Goal**: Record ledger events for the real product actions that consume limits or support usage review.
**Depends on**: Phase 208.
**Requirements**: USAGE-03
**Success Criteria**:

1. Question submission writes a ledger event with the effective entitlement source from v5.6.
2. OCR/AI answer generation writes distinguishable events when those actions run.
3. Teacher-help request writes an event without double-counting retries.
4. Chat and hint usage write events using the same service contract.
5. Instrumentation failures are handled so they do not silently corrupt quota state; behavior is documented.

### Phase 210: Quota Reconciliation And Admin Usage Query

**Goal**: Keep existing counters stable while making ledger/counter agreement inspectable.
**Depends on**: Phase 209.
**Requirements**: USAGE-04
**Success Criteria**:

1. Current daily question counter remains the enforcement path unless explicitly changed.
2. Reconciliation compares counter totals and ledger totals for a bounded period.
3. Admin usage query can filter by parent, student/user, period, product area, and entitlement source.
4. Response shape is support-safe and does not expose raw internal-only metadata by default.
5. Tests cover ledger/counter agreement, mismatch reporting, and admin query boundaries.

### Phase 211: v5.7 Usage Ledger Release Gate

**Goal**: Close v5.7 as a complete usage instrumentation milestone.
**Depends on**: Phase 210.
**Requirements**: VERIFY-40
**Success Criteria**:

1. Usage ledger contract, repository/service, instrumentation, reconciliation, and admin query are complete.
2. Focused backend tests pass for idempotency, quota compatibility, and query response shape.
3. Documentation records any product areas intentionally not instrumented.
4. Release evidence includes commit SHA, test command/output summary, and known deferred items.
5. v5.8 verification milestone handoff is updated.

## Acceptance

- Usage events are durably recorded with actor, subject, product area, action, quantity, entitlement source, period, idempotency key, and timestamp.
- Existing daily counters are not broken.
- Admin can query usage by parent/student/user, period, product area, and entitlement source.
- Parent/customer usage summary handoff exists for v5.9 visibility.
- Tests cover event writes, duplicate/idempotent events, quota exhaustion, ledger/counter consistency, and admin query behavior.
