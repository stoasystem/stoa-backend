# Roadmap: v5.13 Payment And Entitlement Production Completion

**Status:** Planned
**Created:** 2026-07-05
**Prior milestone:** v5.12 Curriculum Editor And Content Migration Buildout

## Goal

Make paid access work as a real product flow instead of only backend readiness: checkout/session creation, provider webhook reconciliation, entitlement activation, usage-limit enforcement, admin support visibility, refund/invoice state, and release evidence.

## Why This Is A New Milestone

This is not a continuation of a v5.12 phase. It is a separate business-critical product capability. Prior milestones created local entitlement and payment readiness primitives, but user-facing paid access can still fail if provider callbacks, entitlement reconciliation, frontend state, and support evidence are not connected and verified end to end.

## Product Purpose

- Parents can understand and complete paid access.
- Students receive the correct entitlement after successful payment.
- Failed, pending, refunded, canceled, or mismatched provider events are visible to support/admin.
- Usage limits and entitlement state agree with payment state.
- The team has enough evidence to decide whether live rollout is blocked by provider credentials or by remaining implementation gaps.

## Implementation Strategy

- Start with a code/product reality audit of current checkout, subscription, entitlement, quota, and frontend paywall behavior.
- Implement the smallest complete end-to-end paid access loop before adding provider edge cases.
- Treat provider webhooks as source-of-truth events, with idempotent reconciliation.
- Keep manual admin override available, but make it visibly distinct from provider-backed paid access.
- Add support-safe admin evidence for payment state without exposing raw provider secrets or sensitive payloads.
- Verify locally with deterministic provider fixtures; run live smoke only when credentials and rollout approval exist.

## Workstreams

1. Payment reality audit and contract refresh.
2. Checkout/session and frontend paywall state completion.
3. Webhook reconciliation and entitlement activation.
4. Refund, cancellation, invoice, and failed-payment support state.
5. Usage-limit compatibility and release gate.

## Acceptance

- Successful provider event activates the correct entitlement exactly once.
- Duplicate/out-of-order provider events are idempotent and auditable.
- Failed/pending/refunded/canceled states produce correct user/admin behavior.
- User-visible paid state and backend entitlement state match after reconciliation.
- Admin can inspect support-safe payment and entitlement evidence.
- Tests cover happy path, duplicate webhook, missing provider event, failed payment, cancellation, refund, manual override interaction, and quota enforcement.

## Out Of Scope

- Finance/accounting exports beyond support-safe invoice/refund visibility.
- Switching payment providers.
- Live rollout without approved production provider credentials.
