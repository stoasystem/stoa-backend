# Requirements: v4.9 Production Notification And Native Delivery Rollout

**Milestone:** v4.9
**Status:** Complete
**Created:** 2026-06-13

## Goal

Turn the v4.2 production notification readiness foundation and v3.6 local WebSocket functionality into production-deliverable notification capability: live WebSocket/API Gateway rollout readiness, provider-backed email/push delivery, frontend/native notification UX handoff, native push token registration, and live smoke evidence.

This is an internal development milestone. Prioritize usable notification delivery features and rollout wiring. Keep checks focused on delivery correctness, provider configuration, preference behavior, and non-mutating live smoke; do not broaden this milestone into unrelated security/compliance work.

## Requirements

### PRODNOTIF-01 Production Notification Rollout Contract And Ownership

Implementers have a concrete production notification rollout contract before deployment or provider delivery work expands.

Acceptance criteria:

- Contract identifies backend, frontend, native, infrastructure, and provider ownership boundaries.
- Contract defines live WebSocket/API Gateway route expectations, auth/subscription behavior, fallback behavior, and deployment prerequisites.
- Contract defines provider-backed email digest and push delivery modes, credential/configuration states, preference behavior, and failure states.
- Contract defines notification event categories that are in scope for live rollout and which remain backend-only.
- Contract defines live smoke boundaries, rollout gates, observability evidence, and rollback behavior.

### PRODNOTIF-02 Live WebSocket/API Gateway Deployment Readiness

The backend has concrete readiness for live WebSocket delivery beyond local functional behavior.

Acceptance criteria:

- CDK/infrastructure readiness or documented handoff covers WebSocket routes, connect/disconnect/message handling, Lambda/runtime configuration, and environment variables.
- Backend delivery status can distinguish local-only, configured, deployed, provider-blocked, and live-ready states.
- WebSocket fanout keeps durable notification event fallback behavior when live delivery fails.
- Admin/operator status exposes live endpoint, recent delivery attempts, stale connection cleanup state, and configuration blockers without exposing secrets.
- Tests or focused checks cover configured/unconfigured live WebSocket states, fanout fallback, stale cleanup, and admin status shape.

### PRODNOTIF-03 Provider-Backed Email Digest And Push Delivery

Notification delivery can use configured providers for email digests and push notifications while preserving preferences and fallback behavior.

Acceptance criteria:

- Email digest send path supports approved provider configuration, scheduling/manual trigger readiness, template metadata, recipient selection, and send/refusal/failure evidence.
- Push delivery path supports provider configuration, native token registration readiness, token lifecycle state, preference gating, and send/refusal/failure evidence.
- Delivery decisions honor durable notification preferences and event category/channel rules.
- Provider responses are redacted and persisted as operator-useful delivery evidence.
- Tests cover provider configured/unconfigured states, preference opt-out, digest selection, push token missing, provider failure, and successful delivery fixtures.

### PRODNOTIF-04 Frontend And Native Notification UX Handoff

Frontend/native work has a concrete integration contract for live notifications.

Acceptance criteria:

- Frontend/native handoff defines API routes, WebSocket endpoint discovery, token registration contract, preference UI behavior, notification center refresh behavior, and fallback states.
- Student, parent, tutor, and admin notification UX expectations are documented for live delivery, offline/reconnect, unread/read/archive, and digest/push preference controls.
- Native push token registration includes platform, token hash/reference, lifecycle status, last seen timestamp, and revocation behavior.
- Cross-workspace follow-up points for `/Users/zhdeng/stoa-frontend` and future native apps are explicit.
- Handoff avoids hidden demo fallback for user-critical notification flows.

### VERIFY-32 v4.9 Production Notification Release Gate And Live Smoke

v4.9 closes with focused notification rollout evidence and updated remaining-feature planning.

Acceptance criteria:

- Focused backend tests and relevant static checks pass or isolate documented pre-existing failures.
- Live WebSocket readiness, provider-backed email/push delivery, token registration, preferences, and frontend/native handoff are verified.
- Release evidence records rollout state: local-only, configured, provider-ready, live-smoked, blocked, or deferred.
- Requirements, roadmap, state, feature gap docs, and remaining-feature queue reflect completed v4.9 work.
- Next milestone recommendation is updated from the remaining feature queue.
