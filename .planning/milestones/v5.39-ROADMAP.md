# Roadmap: v5.39 Platform Reliability And Internal Operations Scale

**Status:** Planned
**Created:** 2026-07-07
**Prior milestone:** v5.38 Learning Outcomes Curriculum And AI Quality Scale

## Goal

Prepare STOA for larger controlled cohorts by hardening reliability, observability, data quality, admin operations, teacher/support workflows, incident handling, and release operations.

## Function Purpose

- Reduce operational drag before more families, teachers, and support staff depend on the system.
- Make admin, teacher, support, and release workflows repeatable instead of founder-operated.
- Catch data drift, queue overload, provider degradation, and release regressions before users report them.

## Implementation Strategy

- Use operational evidence from v5.35-v5.38 as input.
- Prioritize high-frequency admin/support/teacher tasks and reliability risks over speculative platform work.
- Add dashboards, reconciliation, runbooks, tests, and release gates where they reduce real incidents or manual work.
- Close with expansion-readiness or operations-remediation decision.

## Phases

- [ ] **Phase 367: Reliability Incident And Data Quality Audit** - Analyze incidents, data drift, queue overload, reconciliation failures, and release regressions from recent milestones.
- [ ] **Phase 368: Admin Teacher And Support Workflow Scale-Up** - Improve internal workflows for account operations, teacher dispatch, support handoff, billing fixes, and content operations.
- [ ] **Phase 369: Observability Dashboard And Alert Tuning** - Tune dashboards, alerts, SLOs, and ownership for real operational signals.
- [ ] **Phase 370: Release Rollback And Migration Discipline** - Strengthen release checklists, rollback, migrations, smoke tests, fixture hygiene, and evidence capture.
- [ ] **Phase 371: Operations Scale Readiness Gate** - Decide whether larger expansion is operationally ready or needs another hardening cycle.
