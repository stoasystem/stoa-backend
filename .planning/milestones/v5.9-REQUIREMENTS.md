# Requirements: v5.9 Parent Admin Operations Visibility

**Milestone:** v5.9
**Status:** Planned
**Created:** 2026-07-03

## Requirements

### OPSVIS-01 Customer Account Operations Summary

Acceptance criteria:

- Parent/customer response includes effective plan, billing status, entitlement source, current-period usage summary, verification state, and available actions.
- Empty, free, paid, pending, past-due, canceled, and manually overridden states are represented.
- Response avoids raw provider/internal details.
- Actions cover manage subscription, verify email, resend verification, quota blocked/contact support, where applicable.
- Backward compatibility with existing parent subscription views is documented.

### OPSVIS-02 Admin Support Operations Detail

Acceptance criteria:

- Admin response includes entitlement source, manual override state, billing provider state, recent provider event, usage summary/recent events, quota state, verification status, and support timestamps.
- Admin filters and lookup paths are documented for parent, student, email, billing customer, and key status fields where feasible.
- Support-safe explanations distinguish customer-facing reason from internal diagnostic reason.
- Bounded admin actions are documented only when backed by existing safe primitives.
- Role/admin-only boundaries are testable.

### OPSVIS-03 Backend Aggregation APIs

Acceptance criteria:

- Customer account summary composes entitlement, billing, usage, and verification data.
- Admin account detail composes the same domains with support-grade detail.
- Aggregation handles missing child, missing billing, missing ledger, pending verification, and no-usage states without 500s.
- Response shapes are stable and frontend-ready.
- Existing billing/admin endpoints remain backward compatible or have explicit migration notes.

### OPSVIS-04 Frontend/Admin Handoff

Acceptance criteria:

- Handoff document or fixtures describe parent and admin UI states.
- API examples cover successful, blocked, pending, unverified, no-data, and manually overridden states.
- Response-shape tests protect frontend contracts.
- Role-boundary tests cover parent-only and admin-only access.
- Remaining frontend gaps are captured as explicit future tasks if not implemented.

### VERIFY-42 v5.9 Core Operations Closeout

Acceptance criteria:

- Focused tests pass.
- Final audit confirms v5.6-v5.9 entitlement, usage, verification, and visibility coverage.
- Release evidence records commit SHAs, test command/output summary, API examples, and deferred items.
- Docs, roadmap, state, stoa_docs gap audit, and next milestone queue are updated.
- Next product milestone recommendation is explicit.
