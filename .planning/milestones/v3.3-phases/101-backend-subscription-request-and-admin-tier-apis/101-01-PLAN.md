# Plan: Phase 101 Backend Subscription Request And Admin Tier APIs

**Status:** Complete
**Milestone:** v3.3 Subscription Operations MVP
**Requirement:** SUBOPS-02

## Tasks

1. Add a backend subscription operations service for request creation, filtering, lifecycle transitions, history events, and explicit tier application.
2. Add parent endpoints:
   - `GET /parents/me/subscription`
   - `POST /parents/me/subscription/requests`
   - `GET /parents/me/subscription/requests`
3. Add admin endpoints:
   - `GET /admin/subscriptions/requests`
   - `GET /admin/subscriptions/requests/{request_id}`
   - `PATCH /admin/subscriptions/requests/{request_id}`
   - `POST /admin/subscriptions/requests/{request_id}/apply`
4. Preserve existing `subscription_tier` semantics so quota behavior continues to use the same profile field.
5. Add focused tests for parent request creation, duplicate open request protection, admin filtering, approve-vs-apply behavior, invalid apply behavior, and role gating.

## Non-Goals

- No Stripe/TWINT integration.
- No payment collection, invoices, refunds, taxes, or webhooks.
- No CDK or DynamoDB index changes.
- No frontend implementation.
