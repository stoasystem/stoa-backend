# Plan 473-15: Provider-coordinate invariants and isolated cleanup recovery

<objective>
Close CR-009 and WR-009 by rejecting malformed provider success before any recovery fence is removed, defending the same invariants in repository transitions, and making scheduled cleanup isolate every candidate while preserving truthful retryable state.
</objective>

## Artifacts this phase produces

- A private helper that accepts only a non-empty non-whitespace string for required `UploadId`, `VersionId`, and the selected required ETag invariant and otherwise raises the existing redacted retryable dependency decision.
- Repository guards on every multipart/staging/immutable provider-success transition that removes an operation fence.
- Cleanup recovery that retains exact never-reused keys/fences/tombstones until the target is durably recovered, proven absent, or exactly deleted.
- A per-candidate scheduled-job isolation boundary and deterministic `CleanupSummary` retryable/deleted/protected/deferred/skipped counts.
- Wave 0 adversarial fixtures for `{}`, missing keys, `None`, non-string, empty, whitespace-only coordinates, stale recovery lookup failure, and later-candidate continuation.

<must_haves>
Truths: malformed provider success cannot become lifecycle success, erase recovery identity, or produce false cleanup truth; one broken cleanup row cannot starve the batch. Artifacts: strict coordinate validation, repository guards, isolated job behavior, and crash/restart regressions. Key links: validation occurs before every fence-removing transition and cleanup records only coordinate-free item categories.
</must_haves>

<threat_model>
ASVS L1 assets are private uploaded bytes, multipart capabilities, exact staging/immutable versions, lifecycle truth, cleanup throughput, and redacted failure behavior. Trust boundaries are provider success mappings to repository transitions, restart reconciliation to destructive cleanup, and one cleanup candidate to the rest of a scheduled batch. High-severity blockers are accepting a false success coordinate, discarding the only recovery fence, deleting a newer/unrelated target, claiming `cleanup_complete` with residual bytes, or aborting the batch on one malformed row; any unresolved blocker stops execution. Mitigations are strict type/non-whitespace validation, repository defense-in-depth before `remove_operation=True`, durable exact-key operation state, exact VersionId deletion, bounded provider enumeration, non-consumable retry tombstones, and a final redacted per-item isolation boundary. Verification injects every malformed mapping shape at issuance, staging completion, and immutable promotion; process restart; lookup/table/provider faults; and multiple candidates. Residual real-S3 and deployed-scheduler behavior remains NOT RUN and Phase 479-owned.
</threat_model>
