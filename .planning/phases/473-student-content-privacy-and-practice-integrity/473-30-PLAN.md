# Plan 473-30: Moderation history ownership, fencing, and purge

<objective>
Close the moderation-derived private-content branch by assigning every summary/history/event row to its authoritative student, fencing all writers on the canonical account generation, and exhaustively scrubbing free text.

Purpose: Prevent moderation notes and history rows—especially legacy ownerless `EVENT#` rows—from surviving or reappearing after account deletion begins.
Output: Owner propagation, fenced writers, resumable moderation purge, notification handoff contract, and focused tests.
</objective>

## Artifacts this phase produces

- A closed inventory of `MODERATION#{case_id}/SUMMARY` and every `EVENT#{...}` writer/field.
- A repository/service contract that propagates the authoritative question owner and exact Plan 29 fence generation.
- An independent `moderation` deletion handler with summary and event cursors, item debt, and quiescence proof.

<must_haves>
Truths: every moderation row has authoritative ownership; all private text is removed; every writer loses to the same fence; notification handoff remains owner-bound; pagination/debt is restartable. Artifacts: source registry, repository/service fences, scrub handler, and focused tests. Key links: question owner to rows, fence to writes, pages to branch state, and moderation to Plan 34 notifications.
</must_haves>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| question -> moderation case/event | Legacy event rows omit student ownership and cannot safely be purged from actor IDs alone. |
| account fence -> moderation writers | Notes and resolution workers may append after a deletion scan. |
| summary/event page -> branch state | Separate pages and inline history can be partially scrubbed or recreated. |
| moderation -> notification | A best-effort notification can create another durable copy without owner/generation context. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-473-30-01 | Information disclosure | summary/history/event rows | mitigate | REMOVE the source-registered private field set and assert a strict tombstone allowlist. |
| T-473-30-02 | Spoofing | legacy event ownership | mitigate | Resolve the case's question through an authoritative strong read and persist owner/generation on every event. |
| T-473-30-03 | Tampering | moderation writers | mitigate | Same-table canonical fence ConditionCheck plus existing-owned-row conditions for updates/appends. |
| T-473-30-04 | Denial of service | paginated purge | mitigate | Persist separate summary/event cursors, per-item debt, progress validation, and zero-discovery epochs. |
| T-473-30-SC | Tampering | package installs | mitigate | No package-manager install is planned; retain the standard package-legitimacy gate if scope changes. |
</threat_model>
