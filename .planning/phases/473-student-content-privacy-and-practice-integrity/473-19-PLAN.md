# Plan 473-19: Exact provider absence and cleanup convergence

<objective>
Make multipart abort, version deletion, and asynchronous cleanup prove exact provider absence across pagination and retries, while preserving the 30-minute intent lifetime, the exact D-15 `upload_expired` contract, and terminal fences per D-06 through D-10 and D-16/D-17.

Purpose: Eliminate false cleanup completion, destructive lease races, and unbounded or truncated provider reconciliation.
Output: Stateful cleanup contracts, durable continuations, exact absence proof, and scrubbed part metadata.
</objective>

## Artifacts this phase produces

- A failing lower-bound cleanup suite with no-op-success and commit-then-raise provider behavior.
- Exact repeated abort/version deletion verification with durable continuation.
- TTL/terminal eligibility and PART-row lifecycle enforcement.

<must_haves>
Truths: API success is not absence proof; cleanup begins only from expired/terminal lifecycle state; every page/candidate can resume. Artifacts: stateful fakes, durable markers, and bounded cleanup logic. Key links: state to claim, provider listing to absence, and progress to next invocation.
</must_haves>

<threat_model>
ASVS L1 assets are live uploads, terminal state truth, exact UploadId/version coordinates, student bytes, cleanup cursors, and private diagnostics. Trust boundaries are scheduler to cleanup job, repository candidate pages to service, and S3 mutation acknowledgements to absence proof. High-severity blockers are abort/delete call-success treated as completion, stale worker deleting an active generation, short-lease expiry causing destruction before intent expiry, malformed/repeating pagination, a single candidate aborting later candidates, orphan PART rows, or leaked coordinates. Mitigate with terminal-generation CAS, exact listing loops, validated bounded markers, durable cursors, per-candidate isolation, mutation/reconciliation budgets, TTL/scrub, and closed telemetry. Deployed scheduling/IaC and real S3 remain NOT RUN for Phase 479.
</threat_model>
