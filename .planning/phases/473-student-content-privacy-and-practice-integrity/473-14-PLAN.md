# Plan 473-14: Immutable source proof and final evidence publication

<objective>
Lock the remediated Plans 473-12 and 473-13 source, execute every retained privacy, authorization, cleanup, replay, dependency, and resource-safety gate on that one immutable candidate, then publish complete redacted evidence for all Phase 473 requirements and decisions.
</objective>

## Artifacts this phase produces

- `docs/security/route-authorization-inventory.json`, generated twice byte-identically and committed before the candidate SHA is fixed.
- `docs/security/phase-473-evidence.md`, with executable-result coverage for CR-007, WR-006/007/008, V9PRIV-01/02/03, and every individual D-01 through D-22.
- `.planning/phases/473-student-content-privacy-and-practice-integrity/473-VALIDATION.md`, recording exact commands, results, tested source SHA, residual boundaries, and honest `NOT RUN` external checks.
- `docs/security/phase-473-evidence-manifest.json`, schema version 1, containing `testedSourceSha` plus reproducible SHA-256 and byte sizes for evidence and validation.

<must_haves>
Truths: all requirements and all twenty-two decisions are proved on one immutable source candidate, no evidence artifact contains a seeded private value, and unavailable external observations remain NOT RUN. Artifacts: deterministic pre-SHA inventory, evidence, validation, and two-artifact manifest. Key links: only a three-path docs commit follows the candidate; manifest hashes reproduce exactly and both narrative artifacts contain the same testedSourceSha.
</must_haves>

<threat_model>
ASVS L1 assets are student content and metadata, ownership boundaries, answer secrecy, upload lifecycle truth, exact object coordinates, internal diagnostics, route authorization policy, and evidence integrity. Trust boundaries are source/test code into executable observations, route discovery into checked inventory, test fixtures into logs and evidence, immutable candidate source into later documentation, and local emulation into external-system claims. High-severity blocking threats are stale or selective evidence, omission of any D-01..D-22 decision, seeded canary leakage in logs/inventories/evidence/validation/manifest, mutation after testing, a manifest bound to different bytes, authorization or cleanup regression, and claiming unobserved S3/deployment/log behavior; any unresolved high-severity threat blocks completion. Mitigations are clean-tree source lock, exact regression commands, exhaustive programmatic decision enumeration, executable-result requirements per decision, double inventory generation and `--check`, fixed-string denylist across every output, docs-only post-SHA diff, artifact hashing, and explicit NOT RUN rows. Verification rejects any missing decision, missing result, mismatched SHA/hash/size, extra post-SHA path, or private value.
</threat_model>
