# Plan 473-12: Crash-safe immutable cleanup and recovery

<objective>
Close CR-007 by making each upload provider mutation recoverable from a durable fenced operation record and by deleting every exact unreferenced staging and promoted immutable target before cleanup may report completion.
</objective>

## Artifacts this phase produces

- Repository operations `prepare_staging_issuance`, `record_staging_multipart`, `claim_staging_assembly`, `recover_staging_completion`, `begin_immutable_promotion`, `record_immutable_version`, `mark_cleanup_multipart_aborted`, `mark_cleanup_staging_deleted`, and `mark_cleanup_immutable_deleted`.
- Upload-intent private fields for unique staging and immutable keys, provider multipart ID when known, operation kind/fence/lease, expected checksum/length, exact returned VersionIds/ETags, and cleanup target progress; none are public model fields.
- Recovery service entry points used by normal completion and scheduled cleanup for stale `issuing`, `assembling`, and immutable-promotion states.
- Cleanup candidate/claim logic covering terminal states, ordinary expiry, and expired operation leases while excluding active upload/consume states.
- Restart, lost-response, split-provider/database, durable-reference, same-key/new-version, and idempotent cleanup fixtures in `tests/test_attachment_security.py` and route-level lifecycle coverage in `tests/test_files.py`.

<must_haves>
Truths: D-09 is true across process death, provider success with lost database response, and cleanup retry. Artifacts: durable operation coordinates, fenced recovery transitions, exact per-target deletion progress, and lifecycle tests. Key links: unique server-only keys are persisted before provider mutation and cleanup cannot complete before every unreferenced exact target is gone.
</must_haves>

<threat_model>
ASVS L1 assets are private student bytes, exact staging and immutable versions, multipart capabilities, durable references, storage quota, and lifecycle truth. Trust boundaries are DynamoDB operation state to S3 mutation, S3 success to repository persistence, scheduled recovery to destructive delete, and bounded reference scans to cleanup completion. High-severity blocking threats are CR-007 orphan retention, deletion of a referenced or newer version, stale-worker completion, upload resurrection, and falsely reporting `cleanup_complete`; any unresolved high-severity path blocks this plan. Crash windows include death after multipart creation, CompleteMultipartUpload, immutable PutObject, exact delete, or abort but before the corresponding repository write. Mitigations are never-reused keys persisted before mutation, operation fences and expiring leases, immediate persistence of returned provider IDs, exact-key recovery for the unavoidable provider-return split, VersionId-bound deletion, bounded resumable reference discovery, conditional per-target progress, and completion only after all required targets are proven absent/deleted. Verification injects restart at every boundary, stale fences, same-key newer versions, durable-reference races, provider retries, and repository-write loss. Residual real-S3 behavior remains NOT RUN and Phase 479-owned; local closure must not claim it.
</threat_model>
