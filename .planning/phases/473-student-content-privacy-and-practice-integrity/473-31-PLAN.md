# Plan 473-31: Report rows, S3 artifacts, and SES deletion closure

<objective>
Close D-10 for every durable report/recovery/audit copy and every real S3/SES producer, including write-before-repository races, lost provider responses, and lawful retention conflicts.

Purpose: Ensure report generation/edit/recovery cannot create an undiscoverable private object or send after deletion begins, and ensure row/object evidence never overclaims absence.
Output: Fenced report writers, exact object/email intents, exhaustive record/artifact purge branches, explicit legal-retention policy state, and lower-bound tests.
</objective>

## Artifacts this phase produces

- A closed report schema/writer registry spanning summary, all child drafts/previews/audits, recovery rows, and report-scoped immutable retention rows.
- Durable owner-partitioned `REPORT_OBJECT#` and `REPORT_EMAIL#` operation state tied to the canonical account generation.
- Independent `report_records` and `report_artifacts` handlers with exact cursor/debt/quiescence evidence.

<must_haves>
Truths: all report/support rows are fenced/scrubbed; object intent precedes each put; lost responses reconcile exact versions; delivery workers recheck; external delivery and legal holds are classified honestly. Artifacts: row registry, object/email intent state, three branch handlers, and lower-bound tests. Key links: producer to intent/provider, worker to send lease, pages/versions to branch state, and policy hold to blocked status.
</must_haves>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| report service -> repository | `store_and_send_weekly_report` and artifact edits currently write private S3 bytes before the report row/fence link. |
| application -> S3 versioned storage | Commit-then-raise, delete markers, pagination, and link rejection can orphan exact versions. |
| recovery/weekly worker -> SES | A queued or already-claimed send can execute after deletion/finalization. |
| deletion policy -> immutable/legal hold | Lawful retention can forbid physical deletion and must not be mislabeled as purged. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-473-31-01 | Information disclosure | report/recovery/audit rows | mitigate | Source-register every family/private field and scrub to strict noncontent tombstones. |
| T-473-31-02 | Repudiation | S3 put/delete | mitigate | Pre-register exact intent, tag operation/checksum, reconcile all versions/markers, and prove exact absence. |
| T-473-31-03 | Tampering | report writers/workers | mitigate | Same-table canonical fence checks on every write/claim/link and existing-row conditions on updates. |
| T-473-31-04 | Information disclosure | SES send | mitigate | Durable deletion-aware send lease with pre-provider recheck and non-retryable accepted/unknown classification. |
| T-473-31-05 | Denial of service | legal hold | accept/mitigate | Minimize legally required fields, retain explicit policy debt, and block purge completion without restoring access. |
| T-473-31-06 | Information disclosure | support/CRM feed duplicates | mitigate | Resolve owner from report/job lineage, fence each primary/feed append, and scrub every denormalized copy. |
| T-473-31-SC | Tampering | package installs | mitigate | No package-manager install is planned; retain the standard package-legitimacy gate if scope changes. |
</threat_model>
