# Plan 473-03: Durable conversation attachment history and reuse

<objective>
Make conversation attachments real durable resources: atomically bind fresh or reused owner files to student messages, retain safe summaries in history, feed bounded internal extraction to AI, and release bytes only when no durable reference remains.
</objective>

## Artifacts this phase produces

- `extract_attachment_text`, `extract_pdf_text`, `extract_docx_text`, `extract_pptx_text`, `extract_xlsx_text`, and `extract_plain_text` in `document_extraction_service.py`.
- `bind_message_attachments`, `list_attachment_summaries`, `release_resource_attachments`, and `purge_student_attachments` service operations.
- Message/attachment association and reference query/delete operations in `attachment_repo.py`.
- `attachments: list[AttachmentSummary]` on `ChatMessage` and both send/history response paths.

<must_haves>
Truths: D-03/D-04/D-10..D-14/D-16/D-17 hold for conversation history and reuse. Artifacts: extraction boundary, association services and response summaries. Key links: one binding path serves regular and streaming sends; extraction never becomes public history metadata.
</must_haves>

<threat_model>
ASVS L1 assets: conversation history, durable files, storage quota, extracted document text and AI prompts. Trust boundaries: `attachmentIds` to owner lookup; saved attachment to new message association; parser output to model; deletion to reference/accounting updates. Threats: foreign ID attachment, duplicate ID quota/ref manipulation, file substitution after validation, parser bomb/active content, extracted text leakage, message written before attachment transaction, stream path bypass, or deletion removing bytes still referenced elsewhere. Mitigations: immutable validated attachment/version, bounded unique ID list, owner/active conditions, transaction before AI, one shared send implementation, internal-only extraction projection, reference-aware release and redaction tests. Verification: normal/stream parity, fresh/reuse/foreign/mixed lists, parser limits, history snapshots, last-reference deletion and multi-reference preservation. Residual risk: sophisticated document malware is rejected/contained structurally but not executed; no active content is processed.
</threat_model>
