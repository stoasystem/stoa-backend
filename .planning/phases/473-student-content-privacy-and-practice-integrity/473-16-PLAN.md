# Plan 473-16: Provider-body ownership and conversation transport convergence

<objective>
Close WR-010 and WR-011 by taking ownership of every returned provider body before inspecting it and by normalizing every conversation command repository transport path into the stable structured retry/replay contract without weakening idempotency, concealment, or zero-effect guarantees.
</objective>

## Artifacts this phase produces

- A literal provider-body ownership boundary shared by staging validation/promotion and immutable conversation extraction.
- Exact-once closure regressions for missing/non-callable/raising `read`, bounded read failure, parser/validation failure, and raising `close`.
- A conversation repository call adapter/classifier covering command lookup, claim transaction, attachment transaction, race/lost-response rereads, replay polling, AI lease, terminal/completion transitions, and stored-message recovery reads.
- Regular and SSE route matrices proving safe retry responses and same-fingerprint convergence after ambiguous committed transport failures.

<must_haves>
Truths: malformed provider bodies never leak connections; connected conversation repository failures never escape the structured API boundary or break exact replay semantics. Artifacts: ownership helper/pattern, repository translator, and adversarial tests. Key links: cleanup exceptions cannot replace the primary result and ambiguous transaction failures reread the command before deciding whether to replay or return retry guidance.
</must_haves>

<threat_model>
ASVS L1 assets are provider connection capacity, immutable attachment bytes, conversation content, command fingerprints, quota, durable messages/associations, hidden ownership, and internal diagnostics. Trust boundaries are provider response mapping to Body lifetime, repository/SDK transport to public API, and ambiguous transaction response to retry/replay. High-severity blockers are a leaked closable body, raw repository/provider exception disclosure, duplicate quota/message/attachment/AI effects, changed-content replay acceptance, foreign-ID enumeration, or treating a committed lost response as an uncommitted retry; any unresolved blocker stops this plan. Mitigations are immediate `try/finally` ownership, best-effort close, closed dependency taxonomy, command-state reread under the same fingerprint, deterministic IDs, atomic transactions, bounded polling, server-owned correlation, and regular/SSE parity. Verification uses property-raising Body spies, generic non-ClientError transport faults before and after commit, synchronized retry, content/type/order mismatch, and canary denial. Residual production connection-pool behavior and deployed logs remain NOT RUN and Phase 480-owned.
</threat_model>
