# Plan 473-04: Atomic question attachment consumption and OCR privacy

<objective>
Remove caller-controlled S3 keys from question submission and make OCR operate only on a validated owner upload or active saved image whose association with the question is conditionally durable.
</objective>

## Artifacts this phase produces

- `QuestionAttachmentInput` and safe `attachment` response field on question models.
- `reserve_question_attachment`, `commit_question_with_attachment`, and `release_question_attachment_reservation` service/repository operations.
- `extract_text_from_attachment` OCR boundary that accepts a resolved internal attachment object rather than bucket/key strings from routers.

<must_haves>
Truths: D-01/D-02/D-07/D-08/D-12..D-17 govern the full question/OCR path. Artifacts: keyless request, atomic association, resolved OCR boundary. Key links: idempotency and public responses use attachment IDs/summaries while storage coordinates remain internal.
</must_haves>

<threat_model>
ASVS L1 assets: student image, OCR text, question content, attachment/quota state and idempotency. Trust boundaries: question JSON to attachment resolver; validated attachment to Rekognition; OCR output to question/AI; transaction outcome to response. Threats: foreign/raw key OCR, TOCTOU between validation and consumption, one upload attached twice, saved non-image used for OCR, idempotency key rebound to another file, OCR failure consuming intent, raw OCR/key leak or question written without association. Mitigations: remove key input, owner/active/type reservation, conditional transaction, exact attachment identity in idempotency, release on transient failure, terminal invalid rules, safe response projection and mutation sentinels. Verification: owner/foreign/missing/expired/reused/race matrix, OCR spies, failure injection and canary scans. Residual risk: Phase 475 still owns daily quota/ledger/question all-side-effect convergence; evidence must state that boundary.
</threat_model>
