# Plan 473-35: Source-sealed private-store registry and account deletion finalizer

<objective>
Seal D-10 against future and already-missed durable copies by source-discovering every private mutation, binding the resulting exact 17-branch registry to deletion commands, and making this plan the only aggregate finalizer.

Purpose: Replace manually curated “probably complete” deletion with a deterministic closed-universe proof that fails on a new private sink, stale branch, unresolved provider debt, or retention exception.
Output: Checked store/writer inventory, retained-evidence policy, mutation/runtime gate, and exact-once terminal finalizer.
</objective>

## Artifacts this phase produces

- `phase-473-private-store-inventory.json` with one source-bound row per durable private effect and one reviewed row per explicit exclusion.
- `phase-473-retained-evidence-policy.json` for content-free accounting/security evidence only.
- A finalizer sealed to the checked inventory digest and exact 17 branch IDs.

<must_haves>
Truths: every mutation is classified; exactly 17 source-backed branches are required; each mapping has owner/fence/purge proof; synthetic writes fail; evidence retention is narrow; finalization requires exhaustive current-generation zero debt. Artifacts: generator/inventories, mutation/runtime tests, and finalizer. Key links: source to mapping, inventory to command, branch proof to terminal fence, and evidence fields to policy.
</must_haves>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| source code -> checked inventory | A new write inside an old module can create an unpurged copy while a manual list remains unchanged. |
| branch result -> aggregate finalizer | Missing, duplicated, stale, or shallow branch results can be mistaken for completion. |
| retained evidence label -> raw row/log | “Accounting” or “security” can become a loophole for private content/hashes without field/TTL enforcement. |
| provider/legal policy -> completion claim | Accepted external delivery and legal holds require different states from backend absence. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-473-35-01 | Information disclosure | unregistered write sink | mitigate | AST/source-discover every mutation and fail one-to-one mapping/exclusion cardinality. |
| T-473-35-02 | Spoofing | branch registry/result | mitigate | Bind inventory digest, exact branch IDs, generation, handler version, subfamilies, and zero epochs. |
| T-473-35-03 | Tampering | digest regeneration | mitigate | Re-run semantic sink rules and isolated synthetic mutations; digest refresh cannot bless an unmapped write. |
| T-473-35-04 | Information disclosure | evidence retention | mitigate | Checked field allowlist, purpose/legal basis, TTL, access, no-private-value scans, and category-only logs. |
| T-473-35-05 | Repudiation | external/legal exception | mitigate | Distinct accepted-external receipt and legal-retention-blocked states; neither counts as content purged. |
| T-473-35-SC | Tampering | package installs | mitigate | No package-manager install is planned; retain the standard package-legitimacy gate if scope changes. |
</threat_model>
