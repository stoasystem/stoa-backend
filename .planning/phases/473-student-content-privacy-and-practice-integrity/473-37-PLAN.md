# Plan 473-37: Crash-safe delivery-intent state machine

<objective>
Close WR-03 by separating safely recoverable pre-effect claims from ambiguous provider-effect state and binding every transition to an opaque versioned intent claim per D-16 and D-17.

Purpose: Let delivery work recover after a worker crash without retaining permanent deletion debt, while ensuring recovery can never duplicate an email, push, or WebSocket effect whose outcome may already be external.
Output: Scope/claim contracts, explicit-time takeover, pre-effect and inflight states, ambiguity terminalization, and crash-state tests shared by Plan 473-38.
</objective>

## Artifacts this phase produces

- New `DeliveryIntentScope` symbol with closed `private_owner` and `global_nonprivate` variants; private scope requires owner/generation, global scope requires a persisted classification seal.
- New `DeliveryIntentClaim` symbol carrying operation ID, opaque lease owner, intent version, expiry, scope digest, and payload digest but no payload/token/endpoint/private identifier.
- New persisted fields `scope_kind`, `scope_digest`, `intent_version`, `effect_state`, `lease_owner`, `lease_expires_at`, `effect_started_at`, and terminal `outcome_status`.
- New repository operations `claim_delivery_intent(..., now_epoch)`, `begin_delivery_effect(...)`, `recover_delivery_intent(...)`, and claim/version-bound `complete_delivery_intent(...)`.
- New `tests/test_phase473_delivery_intent_recovery.py` covering all three required crash positions and stale takeover.

<must_haves>
Truths: only pre-effect work is recoverable, inflight work is ambiguity-terminal, stale claims lose, explicit current time governs takeover, and both scopes share the primitive. Artifacts: scope/claim types, durable states, transition operations, and crash tests. Key links: claim to begin CAS, begin CAS to provider, and durable ambiguous state to replay refusal.
</must_haves>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| durable registered intent -> worker claim | Crashed or concurrent workers may race for one logical provider operation. |
| pre-effect claim -> provider mutation | A crash boundary must distinguish definitely-not-started from possibly-started effects. |
| provider response -> durable outcome | Success response or persistence acknowledgement may be lost. |
| retry/recovery -> provider | Blind retry can duplicate private delivery after an ambiguous first effect. |
| delivery scope -> account fence/global seal | Global classification must never become a bypass for private payloads. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-473-37-01 | Spoofing | delivery claim | mitigate | Opaque lease owner plus intent version/scope/payload digests on every transition. |
| T-473-37-02 | Tampering | expired takeover | mitigate | Compare stored expiry with explicit current epoch and permit takeover only in pre-effect state. |
| T-473-37-03 | Repudiation | provider ambiguity | mitigate | Durable `effect_inflight` marker immediately before invocation; crash/lost response terminalizes as acceptance unknown. |
| T-473-37-04 | Information disclosure | retry path | mitigate | Never retry an inflight/accepted/unknown operation and store only digests/status/times. |
| T-473-37-05 | Elevation of privilege | global scope | mitigate | Require exact persisted classification seal; Plan 473-38 supplies authoritative resolution and source validation. |
| T-473-37-SC | Tampering | package installs | mitigate | No package-manager install is planned; retain the package-legitimacy gate if scope changes. |
</threat_model>
