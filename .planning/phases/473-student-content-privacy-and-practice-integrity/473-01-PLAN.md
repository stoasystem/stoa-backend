# Plan 473-01: Security contracts and Wave 0 privacy fixtures

<objective>
Lock the upload/attachment lifecycle, storage-tier, structured-error, practice-preview and attempt-result contracts before any router migration, and create the adversarial fixtures every later plan must satisfy.
</objective>

## Artifacts this phase produces

- `UploadPurpose`, `UploadStatus`, `AttachmentStatus`, `UploadIntentRequest`, `UploadIntentResponse`, `FinalizeUploadResponse`, `AttachmentReference`, and `AttachmentSummary` in `src/stoa/models/attachment.py`.
- `AttachmentErrorCode`, `AttachmentClientAction`, `AttachmentDecisionError`, `safe_attachment_error_body`, and `attachment_http_response` in `src/stoa/security/attachment_errors.py`.
- `PracticeChallengePreview`, `PracticeHintResponse`, `PracticeAttemptResult`, and `PrivilegedPracticeAnswer` in `src/stoa/models/practice.py`.
- `attachmentStorageBytes` in the effective entitlement `limits` projection.
- Wave 0 adversarial fixture modules `tests/test_attachment_security.py` and `tests/test_practice_privacy.py`.

<must_haves>
Truths: D-13..D-20 are executable contracts, not prose. Artifacts: typed models/error registry/test matrices. Key links: later services and routers import these contracts rather than recreate dictionaries or raw HTTP errors.
</must_haves>

<threat_model>
ASVS L1 assets: student attachment ownership, storage/provider coordinates, OCR/document content, answer keys, explanations, attempt identity and client recovery behavior. Trust boundaries: client JSON to Pydantic models; service failures to public errors; curriculum records to student/result projections. Threats: client-selected owner/key, extra-field smuggling, missing-vs-foreign enumeration, raw exception leakage, preview model gaining an answer field, result returned without an attempt ID, paid tier spoofing, or a new error code lacking a bounded client action. Mitigations: `extra="forbid"`, closed enums, server-generated opaque IDs, exhaustive error/action registry, safe body allowlist, typed preview/result models, authoritative entitlement projection, and negative schema tests. Verification: model validation, registry exhaustiveness, serialization field sets and seeded canary redaction. Residual risk: later router code can bypass models; Plans 02–07 close that integration risk.
</threat_model>
