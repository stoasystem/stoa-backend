# Plan 473-28: Checked immutable evidence capture and publication

<objective>
Capture every local Phase 473 gate as machine-checked receipts on one immutable candidate, then publish evidence whose commands, counts, coverage, privacy, ancestry, and hashes are independently revalidated.

Purpose: Eliminate selective console claims, hidden skips/XPASS, stale hashes, incomplete boundary tables, and moving-source ambiguity.
Output: Strict pytest guard, capture/verifier tooling, checked results JSON, evidence/validation, and non-circular manifest.
</objective>

## Artifacts this phase produces

- An opt-in pytest plugin that makes skip, XFAIL, and XPASS fatal and writes exact node outcomes.
- A stdlib gate runner/verifier with raw logs, per-gate JUnit, exact receipts, and deterministic coverage checks.
- A four-artifact docs-only publication validated before and after commit.

<must_haves>
Truths: no local gate is skipped or weakly counted; every result is source-bound and hash-checked; every decision/requirement/boundary is exact; publication bytes/ancestry reproduce; external limitations are unclaimed. Artifacts: guard, verifier/tests/denylist, results, evidence, validation, manifest. Key links: command to receipt, node to coverage, results to narrative, bytes to manifest.
</must_haves>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| source worktree -> immutable candidate | Uncommitted or changing code can invalidate all observations. |
| subprocess/test runner -> receipt | Exit codes, skips, XPASS, counts, logs, or node identities can be omitted or misreported. |
| checked boundary inventory -> coverage mapping | Inventory entries can be missing, duplicated, or mapped to nodes that never ran. |
| raw receipts -> evidence publication | Narrative tables or hashes can drift from observed machine data. |
| local fakes -> external claims | Local tests cannot prove real S3, deployed scheduler, or production-log behavior. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-473-28-01 | Tampering | candidate/gate execution | mitigate | Record and compare HEAD plus clean status before/after every exact argv; invalidate the whole capture on drift. |
| T-473-28-02 | Repudiation | pytest outcomes | mitigate | Use strict opt-in plugin, `xfail_strict=true`, JUnit, node manifest, and fatal zero-tolerance skip/XFAIL/XPASS counts. |
| T-473-28-03 | Spoofing | coverage tables | mitigate | Exact-set/cardinality compare requirements, decisions, boundary IDs, selectors, and observed node IDs. |
| T-473-28-04 | Information disclosure | logs/results/evidence | mitigate | Fixed denylist digest/count per receipt and fail publication on any match without echoing matched private values. |
| T-473-28-05 | Tampering | docs/manifest ancestry | mitigate | Hash final result/evidence/validation bytes, require a direct exact four-path docs child, and exclude the manifest from its own hash set. |
| T-473-28-SC | Tampering | package installs | mitigate | No package-manager install is planned; retain the standard package-legitimacy gate if scope changes. |
</threat_model>
