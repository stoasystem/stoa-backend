# Plan 473-36: Lease-fenced deletion proof and conflict-safe profile scrub

<objective>
Close CR-01, WR-01, and WR-02 by making the 17-branch account-deletion proof single-worker under an explicit current-time lease, durable per-write CAS, valid production timestamps, and conflict-safe parent-profile scrubbing per D-10 and D-17.

Purpose: Prevent stale workers or stale scanned rows from corrupting deletion evidence, duplicating irreversible cleanup, terminalizing an incomplete proof, or overwriting an active parent's concurrent profile update.
Output: Opaque deletion claims, claim/version/digest-bound persistence and finalization, validated UTC lifecycle time, row-CAS profile scrub, and focused race tests.
</objective>

## Artifacts this phase produces

- New `DeletionCommandClaim` and `DeletionCommandClaimLost` symbols carrying only command ID, generation, opaque lease owner, claimed command version, lease expiry, and durable branch-result digest.
- New/changed repository contracts `claim_deletion_command(..., now_epoch, lease_expires_at)`, `renew_deletion_command_claim(...)`, `persist_branch_result(..., claim, expected_branch_results_digest)`, and claim-bound `finalize_account_deletion(...)`.
- New persisted fields `command_version`, `branch_results_digest`, `lease_owner`, `lease_expires_at`, and per-result `result_version`; no private content enters these fields.
- New `_valid_lifecycle_timestamp` boundary and a timezone-aware UTC default clock for `AccountDeletionService`.
- New `tests/test_phase473_account_deletion_claim_fencing.py` covering two workers, expiry takeover, stale writes/finalization, production time, durable result proof, and parent-row CAS.

<must_haves>
Truths: active claims cannot be stolen, stale workers cannot persist or finalize, durable branch evidence is exact, production time is valid, and concurrent parent changes are not lost. Artifacts: claim contract, CAS/digest operations, time validator, profile CAS, and race tests. Key links: current time to claim, claim to branch write, durable digest to final transaction, and scanned row version to narrow scrub.
</must_haves>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| scheduled/background worker -> command claim | Concurrent scanners and route-trigger continuation may observe the same running command. |
| worker memory -> durable branch proof | A stale in-memory cursor/result set must not replace a newer worker's evidence. |
| durable branch proof -> terminal fence transaction | Completion must reflect the exact current 17-branch rows and current claim. |
| mixed account scan -> active parent profile | The deleting child is private debt, but unrelated parent fields remain actively mutable. |
| application clock -> audit/fence persistence | Blank, naive, or malformed time destroys lifecycle evidence. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-473-36-01 | Spoofing | deletion lease | mitigate | Compare stored expiry with explicit current epoch and require one opaque claim owner/version on every mutation. |
| T-473-36-02 | Tampering | branch results | mitigate | Advance command version and canonical branch-result digest through conditional writes; stale writes fail closed. |
| T-473-36-03 | Repudiation | terminal lifecycle time | mitigate | Generate timezone-aware UTC ISO time in production and validate it before repository persistence. |
| T-473-36-04 | Tampering | parent profile scrub | mitigate | Narrow update or original row-version/digest CAS; conflict remains branch debt and forces a rescan. |
| T-473-36-05 | Elevation of privilege | finalizer | mitigate | Load current command consistently and condition terminal transaction on current claim/version/digest/exact seal. |
| T-473-36-SC | Tampering | package installs | mitigate | No package-manager install is planned; retain the package-legitimacy gate if scope changes. |
</threat_model>
