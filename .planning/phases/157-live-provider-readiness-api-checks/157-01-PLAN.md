# Plan 157-01: Implement Live Provider Readiness API Checks

<objective>
Add an admin-only provider readiness endpoint that verifies live Stripe/TWINT setup without creating real customer charges or provider mutations.
</objective>

<tasks>

## Task 1: Add readiness configuration and provider lookup seams

Type: code

Files:
- `src/stoa/config.py`
- `src/stoa/services/subscription_service.py`

Action:
- Add a webhook endpoint URL setting used only for readiness reporting.
- Add non-mutating provider adapter seams for Stripe account capability lookup and Stripe price lookup.
- Normalize provider failures into blocker states without leaking raw provider exceptions or payloads.

Verify:
- Static inspection confirms no mutating Stripe calls are introduced.

Acceptance criteria:
- Readiness can retrieve `twint_payments` capability status and Standard/Premium price metadata when live credentials are configured.
- Provider failures produce redacted `provider_api_failed` evidence.

## Task 2: Build provider readiness response

Type: code

Files:
- `src/stoa/services/subscription_service.py`

Action:
- Add `get_provider_readiness(settings)` using existing `get_billing_readiness(settings)` as the local foundation.
- Return credential, price, TWINT, webhook, refund, finance, and rollout sections.
- Include TWINT production constraints from Phase 156.
- Fail closed on missing/test/provider-failed states.

Verify:
- Response never includes raw API keys, webhook secrets, or raw provider payloads.

Acceptance criteria:
- Response covers credential mode, price mapping, TWINT eligibility/capability, webhook readiness, refund capability, and accounting metadata.
- Blockers are actionable and safe for admin display.

## Task 3: Expose admin endpoint

Type: code

Files:
- `src/stoa/routers/admin.py`

Action:
- Add `GET /admin/subscriptions/billing/provider-readiness`.
- Require admin role through existing route dependency patterns.
- Return the readiness payload as a typed response.

Verify:
- Non-admin access remains forbidden by existing admin dependency behavior.

Acceptance criteria:
- Admin can retrieve readiness without specifying a parent ID.

## Task 4: Add focused tests

Type: test

Files:
- `tests/test_subscription_operations.py`

Action:
- Cover missing production config.
- Cover production test-key refusal.
- Cover live-ready-but-blocked with TWINT pending or inactive.
- Cover provider API failure redaction.
- Cover readiness success with CHF recurring prices and active TWINT capability.

Verify:
- Run focused subscription operation tests.

Acceptance criteria:
- Tests prove no secrets/raw provider payloads are returned.
- Tests prove readiness distinguishes blocker states required by PAYACT-02.
