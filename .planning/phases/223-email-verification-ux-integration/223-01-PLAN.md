# Plan 223-01: Email Verification UX Integration

## Goal

Make frontend auth flows consume the v5.8 email verification backend contract without treating pending verification as an authenticated session.

## Scope

- Frontend auth API types and calls.
- Auth mutation side effects.
- Shared verification UI.
- Register/login integration.
- Focused frontend e2e coverage.
- Phase verification and progress docs.

## Tasks

1. Add frontend auth contract types.
   - Add `EmailVerificationResponse`, verification request types, and a structured API error helper.
   - Add `resendEmailVerification` and `confirmEmailVerification`.
   - Preserve backend detail codes for `email_verification_required`, expired code, invalid code, and rate-limit states.

2. Update registration flow.
   - Ensure pending email verification clears auth and does not redirect as logged-in.
   - Render a verification panel after registration when `emailVerificationRequired` or `onboardingStatus=email_verification_required` is present.
   - Preserve tutor `pending_review` behavior.

3. Update login flow.
   - Detect `email_verification_required` errors.
   - Render a verification action panel from login without losing the entered email.
   - Keep generic login errors sanitized.

4. Add reusable verification UI.
   - Confirm code form with one-time-code autocomplete.
   - Resend action with sent/already-requested/already-verified/rate-limited messaging.
   - Confirm success state that guides the user back to login.

5. Add focused test coverage.
   - Extend or add auth e2e tests with intercepted backend responses.
   - Cover register pending, login blocked, resend, confirm, invalid/expired/rate-limited states.

6. Verify and document.
   - Run targeted frontend tests and build/lint as feasible.
   - Write `223-VERIFICATION.md` and `223-01-SUMMARY.md`.
   - Update v5.10 roadmap, requirements, and state for Phase 223 completion.
