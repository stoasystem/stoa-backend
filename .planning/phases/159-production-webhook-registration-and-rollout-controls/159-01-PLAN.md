# Plan 159-01: Implement Webhook Readiness And Rollout Controls

<objective>
Add production webhook readiness evidence and persisted rollout controls that independently gate live checkout and direct refund execution.
</objective>

<tasks>

## Task 1: Add rollout control persistence

Type: code

Files:
- `src/stoa/services/subscription_service.py`

Action:
- Add persisted payment rollout control record.
- Support checkout and refund states: `disabled`, `canary`, `enabled`, `rolled_back`.
- Include reason, operator, timestamps, and derived activation state.

Acceptance criteria:
- Effective controls fall back to static config when no persisted record exists.

## Task 2: Wire controls into live-changing operations

Type: code

Files:
- `src/stoa/services/subscription_service.py`

Action:
- Make checkout readiness and direct refund execution use effective rollout controls.
- Treat `enabled` as allowed, `canary` as visible but blocked for general operations, and `disabled`/`rolled_back` as blocked.

Acceptance criteria:
- Rollback prevents new checkout/refunds without hiding existing billing state.

## Task 3: Expose admin controls and webhook readiness

Type: code

Files:
- `src/stoa/routers/admin.py`

Action:
- Add `GET /admin/subscriptions/billing/rollout-controls`.
- Add `PATCH /admin/subscriptions/billing/rollout-controls`.
- Ensure provider readiness response includes effective rollout and webhook readiness details.

Acceptance criteria:
- Admin can inspect and update rollout controls without secrets.

## Task 4: Add focused tests

Type: test

Files:
- `tests/test_subscription_operations.py`

Action:
- Cover default rollout state from config.
- Cover enabling/disabling checkout and refunds independently.
- Cover rollback blocking new refund execution while preserving billing export.
- Cover webhook readiness HTTPS/signing/event/last-observed evidence.

Acceptance criteria:
- Focused subscription operation tests pass.

</tasks>
