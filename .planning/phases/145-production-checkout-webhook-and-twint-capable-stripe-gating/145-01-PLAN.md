# Plan 145-01: Implement Production Checkout, Webhook, And TWINT-Capable Stripe Gating

## Goal

Implement production-ready Stripe checkout and webhook primitives with explicit rollout gating, operator-visible readiness, and TWINT-capable Swiss/CHF subscription behavior.

## Work Items

1. Inspect current subscription service, billing router, parent/admin billing routes, settings, and tests.
2. Add a Stripe gateway abstraction around SDK calls and webhook verification.
3. Add billing readiness computation for `not_configured`, `test`, `live_ready_but_blocked`, and `live_enabled`.
4. Update checkout creation to use provider-backed sessions when configuration allows it and fail closed otherwise.
5. Add TWINT-capable Checkout configuration for eligible Swiss/CHF subscription flows where Stripe account capability and rollout gates permit it.
6. Persist provider mode, event ID, event type, idempotency result, request/correlation IDs, and processing outcome for webhook events.
7. Add provider lookup rows for key Stripe identifiers to avoid broad billing summary scans.
8. Expose redacted admin readiness/status fields without provider secrets.
9. Add focused tests for readiness gates, test/live mismatch, checkout blocking, webhook idempotency, bad signatures, and TWINT eligibility.

## Verification

- `pytest tests/test_subscription_operations.py`
- Relevant broader backend tests if shared subscription behavior changes.
- `ruff` or the repository's focused static check for touched files.

## Non-Goals

- Real customer charges.
- Refund/invoice/dunning implementation beyond fields required for event capture.
- Separate non-Stripe TWINT provider integration.
- Broad CRM, accounting, or marketing automation.

## Output

- Backend code changes for checkout/webhook/readiness.
- Updated tests.
- `145-CHECKOUT-WEBHOOK-TWINT-GATING-SPEC.md` kept aligned with implementation.
- `145-VERIFICATION.md` with command evidence and remaining blockers.
