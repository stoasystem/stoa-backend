# Plan 163-01: Implement Retry Workers And Two-Way Support Ticket Synchronization

<objective>
Add bounded retry behavior for failed third-party support deliveries and provider-neutral ticket status synchronization without importing raw provider payloads.
</objective>

<tasks>
<task id="1" type="auto">
  <name>Add retry state updates</name>
  <files>
    - src/stoa/db/repositories/report_repo.py
    - src/stoa/services/support_destination_service.py
    - src/stoa/routers/admin.py
  </files>
  <action>
    Add admin-only retry behavior for failed third-party support deliveries with attempt count, max attempts, retry exhaustion, next eligible timestamp, and redacted result metadata.
  </action>
  <acceptance_criteria>
    - Failed provider deliveries can be retried until attempt limit is reached.
    - Refused, delivered, internal queue, and missing delivery records are not retryable.
    - Retry updates are persisted and audited.
  </acceptance_criteria>
</task>

<task id="2" type="auto">
  <name>Add provider sync normalization</name>
  <files>
    - src/stoa/db/repositories/report_repo.py
    - src/stoa/services/support_destination_service.py
    - src/stoa/routers/admin.py
  </files>
  <action>
    Add provider-neutral sync behavior that stores provider event IDs, normalized lifecycle status, provider updated timestamp, last synced timestamp, sync freshness, safe provider labels, and conflict markers without raw payloads.
  </action>
  <acceptance_criteria>
    - Duplicate provider event IDs are ignored.
    - Stale provider updates are refused without overwriting newer state.
    - Unknown or conflicting statuses surface `sync_conflict`.
  </acceptance_criteria>
</task>

<task id="3" type="auto">
  <name>Add focused retry and sync tests</name>
  <files>
    - tests/test_admin_report_ops.py
  </files>
  <action>
    Cover retry success/exhaustion/refusal and provider sync success/duplicate/stale/conflict behavior.
  </action>
  <acceptance_criteria>
    - Tests prove metadata-only response boundaries remain intact.
    - Tests prove admin-only access is enforced for new endpoints.
  </acceptance_criteria>
</task>
</tasks>

- `./.venv/bin/ruff check src/stoa/db/repositories/report_repo.py src/stoa/routers/admin.py src/stoa/services/support_destination_service.py tests/test_admin_report_ops.py`

<success_criteria>
- Failed provider deliveries are retryable with attempt limits, backoff metadata, and operator-visible status.
- Provider ticket state can be synchronized through webhook or polling-shaped adapters without importing raw private payloads.
- Stale updates, duplicates, and status conflicts are detected and surfaced.
- Admin support queue/detail views expose retry eligibility, sync freshness, provider state, and conflict markers.
</success_criteria>
