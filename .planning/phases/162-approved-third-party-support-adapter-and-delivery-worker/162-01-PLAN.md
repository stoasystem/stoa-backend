# Plan 162-01: Implement Approved Support Adapter And Delivery Worker

<objective>
Add a controlled `third_party_support` provider delivery path for support-safe evidence packages. The path must fail closed without approval or credentials, create provider-ticket-shaped metadata only when approved/configured, preserve internal queue behavior, and cover success/refusal/failure/idempotency with focused backend tests.
</objective>

<tasks>
<task id="1" type="auto">
  <name>Add provider readiness settings and constants</name>
  <files>
    - src/stoa/config.py
    - src/stoa/services/support_destination_service.py
  </files>
  <action>
    Add settings for third-party support provider approval/readiness and constants for `third_party_support`. Keep existing contract-defined refused modes unless explicitly replaced by the new generic mode.
  </action>
  <acceptance_criteria>
    - `Settings()` defaults keep provider writes disabled.
    - `third_party_support` is recognized as a contract-defined delivery mode.
    - Existing `internal_queue` constants remain unchanged.
  </acceptance_criteria>
</task>

<task id="2" type="auto">
  <name>Implement approved provider adapter delivery</name>
  <files>
    - src/stoa/services/support_destination_service.py
    - src/stoa/routers/admin.py
  </files>
  <action>
    Route `destination_mode=third_party_support` through support package creation and a provider adapter delivery function. Persist provider ticket ID/reference, safe URL, readiness state, lifecycle status, redacted result/error metadata, payload digest, idempotency key, and retry eligibility. Refuse without evidence reads when approval is missing; refuse after package validation if privacy/package validation fails; fail with redacted metadata on provider failure.
  </action>
  <acceptance_criteria>
    - Approved/configured provider delivery returns `delivered` with provider ticket metadata.
    - Missing approval or credentials returns `refused` with no package and no evidence reads where possible.
    - Provider failure persists a metadata-only `delivery_failed`/`failed` record with redacted failure reasons.
    - Duplicate approved delivery returns the existing delivery ID and provider ticket ID.
    - Raw provider payloads and private report markers are not returned or persisted.
  </acceptance_criteria>
</task>

<task id="3" type="auto">
  <name>Add focused support provider tests</name>
  <files>
    - tests/test_admin_report_ops.py
  </files>
  <action>
    Add tests for third-party provider approval/readiness, successful provider ticket creation, missing credentials, unapproved destination refusal, provider failure redaction, privacy validation refusal, duplicate/idempotent delivery, and internal queue continuity.
  </action>
  <acceptance_criteria>
    - Tests prove unknown destination modes still reject before evidence reads.
    - Tests prove `internal_queue` behavior still queues when approved.
    - Tests prove `third_party_support` never exposes private artifact markers or raw credentials.
  </acceptance_criteria>
</task>
</tasks>

- Missing credentials, unapproved destination, validation failure, provider failure, and duplicate delivery are covered by tests.
</success_criteria>
