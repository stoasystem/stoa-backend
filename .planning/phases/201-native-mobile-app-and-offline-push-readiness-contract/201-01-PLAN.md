# Plan 201-01: Define Native Mobile App Offline And Push Readiness Contract

## Goal

Create a concrete v5.6 contract for native mobile role flows, API usage, push/deep-link behavior, offline read-through, and implementation ownership.

## Work Items

1. Confirm the remaining `stoa_docs` feature gaps after v5.5: native apps, live notification delivery, frontend rich curriculum editor, warehouse/BI, external payment/support activation.
2. Define function purpose: native mobile access to core learning, parent progress, teacher dispatch, and notification workflows.
3. Define supported roles, first screens, navigation model, auth/session behavior, and locale handling.
4. Define student assignment/progress/report, parent child progress/report, teacher dispatched queue/detail, and admin/operator mobile surfaces.
5. Define push token lifecycle, notification event types, deep-link targets, read/archive/preference compatibility, and provider-gated no-live-provider behavior.
6. Define offline read-through cache boundaries, stale indicators, reconnect refresh behavior, and mutation limits.
7. Identify backend API gaps, frontend/native ownership, Phase 202 through Phase 205 implementation targets, and release-state labels.

## Verification

- Contract maps every NATIVEAPP-01 acceptance criterion to concrete implementation targets.
- v5.6 scope is buildable during internal development without waiting for payment/support/provider activation.
- Follow-up phases have concrete app shell, push/deep-link, offline UX, and release-gate targets.
