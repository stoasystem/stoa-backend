# Phase 7 Plan 01: Weekly Learning Aggregation Service

## Goal

Create a reusable backend report aggregation service that produces truthful weekly report payloads for linked parent/student/week inputs.

## Tasks

### 1. Add report aggregation service

File:

- `src/stoa/services/report_service.py`

Work:

- Add week-window helpers that accept an ISO week start date or `date` and derive `week_start` / `week_end`.
- Add ISO datetime parsing and week-window filtering helpers.
- Add linked-child lookup helpers using the existing local `parent_id`/`user_id` convention.
- Add conversation listing helper using `GSI-StudentId` and `entity_type=conversation`.
- Add `build_weekly_learning_payload(parent_id, student_id, week_start)` that reads real repositories and returns a compact structured payload.

Acceptance:

- Payload includes parent/student metadata, week window, metrics, weak topics, activity list, and source counts.
- Questions, AI answers, teacher help, practice progress, mistakes, and weak-topic data are represented.

### 2. Preserve truthful empty states

File:

- `src/stoa/services/report_service.py`

Work:

- Ensure empty question/progress/mistake/conversation data returns zero counts and empty arrays.
- Exclude malformed/timestamp-free records from weekly activity counts instead of inventing fallback dates.
- Sort weak topics by frequency then name.
- Sort activity events newest-first with deterministic ID tie-breaks.

Acceptance:

- Sparse data creates a valid deterministic payload.
- No fabricated learning activity appears in the payload.

### 3. Add focused tests

File:

- `tests/test_report_service.py`

Work:

- Test week-window parsing.
- Test empty aggregation produces zero/empty payload.
- Test mixed question/practice/mistake/conversation records produce expected metrics and weak-topic ranking.
- Test unlinked parent/student access raises a clear error.

Acceptance:

- Tests pass locally without real AWS access by monkeypatching repositories/table access.

### 4. Verify

Commands:

- `uv run --extra dev pytest tests/test_report_service.py -q`
- `uv run --extra dev ruff check src/stoa/services/report_service.py tests/test_report_service.py`

## Verification Criteria

- `AGGR-01` is satisfied when the service aggregates all required source types.
- `AGGR-02` is satisfied when empty/sparse source data creates deterministic zero/empty payloads without fabricated activity.

## Risks

- Existing DynamoDB records use mixed timestamp fields; helper logic must be conservative and skip malformed values.
- Parent-child lookup remains scan-based MVP from v1.0; this phase should reuse it rather than introduce a new GSI.
