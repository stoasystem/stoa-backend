# Phase 8 Plan 01: Bedrock Report Generation Service

## Goal

Generate validated parent-facing weekly report content from compact weekly report payloads, with deterministic fallback behavior for failed or unsafe model output.

## Tasks

### 1. Add compact report prompt input

File:

- `src/stoa/services/report_service.py`

Work:

- Add `build_bedrock_report_input(payload)` that extracts only bounded parent/student/week metadata, metrics, weak topics, source counts, and a small activity sample.
- Avoid passing raw full activity text when a short summary is enough.
- Keep input deterministic for tests and reproducible generation.

Acceptance:

- Bedrock input is compact and structured.
- No excessive raw student activity payload is sent.

### 2. Add strict generated report parser and safety validation

File:

- `src/stoa/services/report_service.py`

Work:

- Add a strict JSON parser for generated report output.
- Require `summary`, `strengths`, `weakTopics`, and `recommendations`; allow optional `teacherNote`.
- Normalize string/list fields into a stable report content dictionary.
- Reject parent-facing text containing internal provider, model, or implementation terms.

Acceptance:

- Valid strict JSON is accepted.
- Markdown-wrapped, partial, malformed, missing-field, or unsafe output is rejected.

### 3. Add Bedrock invocation and deterministic fallback

File:

- `src/stoa/services/report_service.py`

Work:

- Add `generate_weekly_report_content(payload, bedrock_client=None)` using the existing Bedrock runtime settings and Anthropic Messages body shape.
- On invoke failure, parse failure, or safety rejection, return deterministic fallback content.
- Ensure fallback content uses truthful metrics and weak-topic data only.

Acceptance:

- Bedrock failures do not prevent report content production.
- Fallback output has the same parent-facing shape as generated output.

### 4. Add focused tests

File:

- `tests/test_report_service.py`

Work:

- Test compact input bounding.
- Test strict parser accepts valid JSON.
- Test malformed output falls back deterministically.
- Test unsafe internal provider/model terms trigger fallback or rejection.
- Test Bedrock invocation uses injected client and expected request shape without real AWS.

Acceptance:

- Tests pass locally without network or AWS credentials.

### 5. Verify

Commands:

- `uv run --extra dev pytest tests/test_report_service.py -q`
- `uv run --extra dev ruff check src/stoa/services/report_service.py tests/test_report_service.py`

## Verification Criteria

- `AI-01` is satisfied when report generation sends compact structured weekly input.
- `AI-02` is satisfied when only strict JSON with the required report sections is accepted.
- `AI-03` is satisfied when malformed or failed Bedrock output returns deterministic fallback content.
- `AI-04` is satisfied when parent-facing copy cannot expose internal provider/model terms.

## Risks

- Parser strictness must not silently accept markdown or prose-wrapped JSON.
- Fallback text must stay truthful for empty weeks and avoid implying activity that did not occur.
