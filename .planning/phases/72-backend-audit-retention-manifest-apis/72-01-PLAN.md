# Plan 72-01: Backend Audit Retention Manifest APIs

## Objective

Add admin-only backend APIs that return metadata-only audit retention status and sealed retention manifests with stable drift-detection metadata, while refusing destructive/WORM actions and preserving the existing privacy boundary.

## Tasks

1. Add repository support for append-only audit retention events and support handoff audit listing.
2. Implement a service that resolves allowlisted audit references, sanitizes evidence, computes canonical digests, builds status/manifest responses, and writes redacted audit metadata.
3. Add admin route request/response models and endpoints for retention status and manifest generation.
4. Add focused backend tests for admin-only gating, schema/digest output, status states, privacy denylist, destructive/WORM refusal, unsupported refs, and audit rows.
5. Run focused lint and pytest for touched backend files.

## Verification

- `AUDITRET-02` and `AUDITRET-03` acceptance criteria are covered.
- Manifest/status responses contain no private artifact markers or secrets.
- Unsupported destructive retention actions are refused before evidence reads.
- Manifest generation records redacted audit metadata only.
