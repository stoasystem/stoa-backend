# Plan: Phase 113 Backend WebSocket Connection And Event Delivery

## Objective

Implement backend WebSocket connection lifecycle, subscription authorization, fallback-safe notification fanout, delivery attempt recording, disconnect cleanup, stale cleanup, and focused tests.

## Tasks

<task id="1" name="Add WebSocket connection repository" type="auto">
Create DynamoDB repository helpers for active WebSocket connection records using the existing single-table style.

Acceptance criteria:

- Repository stores records as `PK=WS_CONN#{connection_id}`, `SK=META`, `entity_type=websocket_connection`.
- Repository supports put/get/update/delete/list operations.
- Repository supports deleting stale records by `expires_at`.
</task>

<task id="2" name="Add WebSocket service lifecycle and authorization" type="auto">
Create service helpers for registering authenticated connections, refreshing heartbeat timestamps, updating subscriptions, disconnect cleanup, channel authorization, and stale cleanup.

Acceptance criteria:

- Connection records include user id, role, subscribed channels, connected timestamp, last-seen timestamp, expiry timestamp, and optional endpoint URL.
- User-specific and role channels are authorized server-side.
- Teacher/tutor role alias behavior is explicit.
- Unauthorized subscription attempts raise HTTP 403.
</task>

<task id="3" name="Add fallback-safe fanout delivery" type="auto">
Layer WebSocket fanout after durable notification creation without making realtime failures destructive.

Acceptance criteria:

- `notification_service.create_event` still writes durable events before realtime fanout.
- Fanout targets only active authorized connections matching user or role recipient channels.
- Delivery envelopes include notification fields plus `deliveryId` and `deliveryAttempt`.
- Missing endpoint, no connections, gone connections, and post failures are non-destructive.
- Delivery attempt metadata is recorded without marking the durable event failed.
</task>

<task id="4" name="Add configuration" type="auto">
Add backend configuration fields needed for WebSocket delivery while keeping local/test execution disabled unless explicitly configured.

Acceptance criteria:

- Settings include WebSocket endpoint and connection TTL controls.
- Empty endpoint prevents live API Gateway post calls while still allowing lifecycle tests.
</task>

<task id="5" name="Add focused backend tests" type="auto">
Add focused tests for connection lifecycle, authorization, fanout, delivery result recording, stale cleanup, and fallback persistence.

Acceptance criteria:

- Tests do not require live AWS WebSocket infrastructure.
- Tests cover direct user delivery and role broadcast delivery.
- Tests prove unauthorized channel subscription is rejected.
- Tests prove durable notification creation survives delivery failures.
</task>

## Success Criteria

- Backend stores active connection records with user id, role, subscribed channels, heartbeat/update timestamps, and expiry.
- Connection/subscription requests are authorized through existing Cognito/JWT-style user claims.
- Existing notification events fan out to active authorized WebSocket connections and record delivery attempts/results.
- Disconnect and stale cleanup keep connection state bounded.
- Focused tests cover lifecycle, authorization, fanout, stale cleanup, and durable fallback behavior.
