# Plan 24-01: Generation Failure Retry

## Objective

Add an admin-only endpoint that retries one `generation_failed` report without touching unrelated reports and records audit metadata for success/failure.

## Tasks

1. Extend admin response models
   - Add a retry response model with report id, status, email status, operation, result, updated time, and artifact availability.

2. Add retry endpoint
   - Add `POST /admin/reports/{parent_id}/{student_id}/{week_start}/retry-generation`.
   - Load the existing report by parent/student/week.
   - Refuse any status other than `generation_failed`.
   - Rebuild payload/content and call `store_and_send_weekly_report`.
   - Record success audit fields after store/send returns.
   - Record failure audit fields and keep `generation_failed` if generation/storage fails.

3. Add tests
   - Successful retry targets only one report and records audit.
   - Retry refuses `email_sent`, `generated`, `email_failed`, and pending statuses.
   - Retry failure preserves `generation_failed` and records error/audit fields.
   - Retry response does not expose raw artifact keys or S3 paths.

4. Verification
   - Run focused pytest and ruff.
   - Write summary and verification artifacts.
   - Update roadmap, requirements, and state for Phase 24 completion.

## Acceptance Criteria

- GEN-01 through GEN-04 are covered.
- The retry endpoint is admin-only.
- Only `generation_failed` is accepted.
- Canonical report identity and artifact key shape are preserved by the reused report service.
- Audit fields are written for success and failure.
