# Phase 9 Plan 01: Report Storage and Email Delivery

## Goal

Persist generated weekly reports and artifacts before emailing parents, while keeping generated reports available when SES delivery fails.

## Tasks

### 1. Add report metadata/status repository support

Files:

- `src/stoa/db/repositories/report_repo.py`

Work:

- Keep `put_report(item)` compatible with existing report reads.
- Add status update support for report id, status, timestamps, and error fields.
- Preserve parent/week GSI fields.

Acceptance:

- DynamoDB metadata can represent generated, emailed, and email-failed reports.
- Existing parent route tests remain compatible with legacy report fields.

### 2. Add artifact rendering and S3 writes

Files:

- `src/stoa/services/report_service.py`

Work:

- Build deterministic report ids and S3 keys from parent/student/week.
- Render full report JSON and simple HTML artifacts.
- Store both artifacts in the private reports bucket before email delivery is attempted.

Acceptance:

- JSON and HTML artifacts are written before SES send.
- Stored metadata includes S3 keys and generated content fields.

### 3. Add SES report email delivery

Files:

- `src/stoa/services/notify_service.py`
- `src/stoa/services/report_service.py`

Work:

- Extend weekly report email sending to accept injected SES client and a generated subject.
- Compose email HTML from stored report content.
- Send only to the linked parent email in the payload.

Acceptance:

- Email includes student name, week range, summary, recommendations, and parent portal link.
- Tests verify destination email is parent email only.

### 4. Preserve report availability on email failure

Files:

- `src/stoa/services/report_service.py`
- `src/stoa/db/repositories/report_repo.py`

Work:

- If SES fails after storage succeeds, update the report item to `email_failed`.
- Include `email_error_class` and `email_error_message` fields.
- Return stored metadata with final status.
- Log identifiers/counts/error class, not full question content.

Acceptance:

- Generated report remains stored and addressable after email failure.
- Status reflects email failure.

### 5. Add focused tests

Files:

- `tests/test_report_service.py`
- `tests/test_parent_children.py` if repository-specific coverage needs extension

Work:

- Test report metadata shape and S3 JSON/HTML writes.
- Test email success status update.
- Test SES failure marks `email_failed` after artifacts are written.
- Test email destination uses linked parent email only.

Acceptance:

- Tests pass locally without real AWS clients.

### 6. Verify

Commands:

- `uv run --extra dev pytest tests/test_report_service.py tests/test_parent_children.py -q`
- `uv run --extra dev ruff check src/stoa/services/report_service.py src/stoa/services/notify_service.py src/stoa/db/repositories/report_repo.py tests/test_report_service.py tests/test_parent_children.py`

## Verification Criteria

- `STORE-01` is satisfied when DynamoDB metadata includes status, stats, summary, recommendations, S3 key(s), timestamps, and error fields.
- `STORE-02` is satisfied when full HTML and JSON artifacts are stored in the private reports bucket.
- `STORE-03` is satisfied when artifacts and metadata are stored before email completion is recorded.
- `EMAIL-01` is satisfied when weekly report email sends only to linked parent email.
- `EMAIL-02` is satisfied when email body includes student name, week range, summary, recommendations, and parent portal link.
- `EMAIL-03` is satisfied when SES failure marks `email_failed` but keeps report storage intact.
- `EMAIL-04` is satisfied when logs avoid full student question content and use identifiers/counts/error classes.

## Risks

- Existing parent route models are legacy-shaped; this phase should preserve legacy fields and defer richer response modeling to Phase 11.
- S3/SES are not called live in tests; fake clients must assert request shape and ordering.
