# Phase 149 Plan: Support Evidence Export Destination Integration

## Goal

Add one approved backend-managed support handoff delivery path, `internal_queue`, while preserving the existing manual `preview`, `copy`, `download`, and `external_write` refusal behavior.

`internal_queue` is not a third-party write. It stores metadata-only delivery/status rows in STOA-owned persistence so Phase 150 can expose operator queue visibility.

## Must Haves

- Preserve `POST /admin/reports/support-handoff-package` response and refusal behavior for existing manual modes.
- Add a sibling admin-only delivery route for `internal_queue`.
- Gate `internal_queue` behind fail-closed runtime config equivalent to `SUPPORT_INTERNAL_QUEUE_APPROVED=true`.
- Validate package privacy before writing any queued/sent delivery record.
- Persist delivery lifecycle records separately from package audit events.
- Store only metadata-safe fields, references, redacted reasons, lifecycle state, idempotency key, and payload digest.
- Use stable idempotency that does not depend on UUID package IDs.
- Keep `external_write`, `shared_mailbox`, `zendesk_ticket`, `freshdesk_ticket`, and `helpscout_conversation` refused/out of scope.

## Out Of Scope

- Third-party provider SDKs or API calls.
- Email/ticket adapter implementation.
- Operator list/detail/retry APIs; Phase 150 owns visibility and retry surfaces.
- Raw package payload persistence, attachments, presigned URLs, S3 keys, authorization headers, cookies, API keys, OAuth tokens, or provider/customer payloads.

### Task 2: Add Delivery Orchestrator Service

**Read first**

- `src/stoa/services/support_handoff_service.py`
- `src/stoa/services/report_audit_retention_service.py`
- `.planning/phases/148-support-destination-contract-and-credential-readiness/148-SUPPORT-DESTINATION-CONTRACT.md`

**Action**

- Add a small service for support destination delivery, preferably `src/stoa/services/support_destination_service.py`.
- Accept an already-built support package plus actor, request/correlation ID, destination mode, and settings/readiness inputs.
- Allow only `internal_queue` for Phase 149.
- Compute a canonical metadata-only payload digest using deterministic JSON serialization and a `sha256:` prefix.
- Compute a stable idempotency key from destination mode plus stable request metadata and evidence references, not from the UUID package ID.
- Build a delivery record with:
  - delivery ID
  - package ID
  - destination mode
  - lifecycle status
  - actor
  - created/updated timestamps
  - correlation ID
  - idempotency key
  - retry count
  - retryable flag
  - provider object reference as internal delivery/status ID
  - redacted refusal/failure reasons
  - privacy result
  - evidence reference IDs
  - payload digest
- Refuse when approval is missing/false, destination is not `internal_queue`, or package privacy/validation fails.
- For approved and privacy-passing `internal_queue`, persist status `queued`.
- Never persist raw package sections or raw outbound bodies.

**Acceptance criteria**

- Successful `internal_queue` delivery returns delivery status `queued`.
- Refused deliveries return delivery status `refused`, do not appear as sent, and include redacted readiness/privacy blockers.
- Duplicate identical requests reuse the same delivery record/idempotency key instead of creating a second queued row.
- Payload/delivery records pass privacy denylist checks.

**Verify**

- `./.venv/bin/pytest -q tests/test_admin_report_ops.py -k "internal_queue and support_handoff"`

## Threat Model

| Threat | Mitigation |
|--------|------------|
| Raw artifact or secret leakage in delivery rows | Store only metadata-safe allowlisted fields and run privacy denylist checks before persistence. |
| Duplicate internal queue entries from retries | Use stable idempotency independent of UUID package ID and reuse existing records. |
| Unauthorized delivery | Keep route admin-only and fail before evidence reads for non-admin or unsupported destinations. |
| Misleading lifecycle status | Keep package validation separate from delivery lifecycle; approval/privacy failures are `refused`, not `queued` or `sent`. |
| Scope creep into third-party writes | Refuse all non-`internal_queue` provider modes in Phase 149. |
