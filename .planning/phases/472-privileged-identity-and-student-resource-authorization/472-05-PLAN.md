# Plan 472-05: Central actor-resource-action-purpose authorization policy

<objective>
Implement the single fail-closed policy and fact loaders used by every identifier-bearing route, including ownership, active bidirectional parent binding, task/assignment-scoped teacher access, purpose capabilities, and incident-only break-glass.
</objective>

## Artifacts this phase produces

- `AuthorizationPolicy.evaluate(actor, authorized_resource, action, purpose)` with deterministic denial semantics.
- Resolver/dependency factories that load a resource once and give the handler that same authorized object.
- Parent, teacher-task/assignment, admin capability and break-glass fact loaders using fresh repository state.
- Parameterized unit/matrix proof including positive controls and handler-mutation sentinels.

## Locked decision coverage

- D-10: parent access requires matching active bidirectional binding and active accounts.
- D-11: teacher access is question/conversation task scope or separate active assignment.
- D-12: admin content access requires exact purpose capability and evidence.
- D-13: revoked binding/assignment denies on the next request.
- D-14: break-glass is short-lived, incident-only, read-only, notified, and reviewed.
- D-15: support lookup exposes bounded metadata only.
- D-16: every identifier route declares resource/action/purpose or fails inventory.
- D-17: student self routes derive the canonical Actor user ID.

<threat_model>
Assets: student profiles, questions, conversations, practice/adaptive state, reports, and relationship existence. Trust boundaries: client identifier to canonical resource; current repositories to policy; policy decision to route handler. Abuse cases: unrelated actor enumerates IDs, one-sided/legacy parent link authorizes, queue visibility broadens teacher access, admin role reads content, revoked fact remains cached, repository outage falls through, handler swaps resource after authorization. Mitigations: load-and-return authorized resource, active bidirectional binding equality, exact current task/assignment scope, purpose capability, no role blanket access, fresh reads/no allow cache, indistinguishable 404, safe 503 before handler, audited break-glass restrictions. Validation: T-472-05, T-472-06, T-472-07.
</threat_model>
