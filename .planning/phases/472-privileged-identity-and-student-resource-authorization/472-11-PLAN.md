# Plan 472-11: Canonical public identity registration and token-bound login

<objective>
Close G-01 by connecting ordinary student/parent registration, confirmation, login, and refresh to the canonical `(issuer, Cognito sub) -> user_id` identity boundary, with durable resumable commands and no email-based business identity selection.
</objective>

## Artifacts this phase produces

- `src/stoa/db/repositories/public_identity_repo.py`: `PublicIdentityCommandState`, `PublicIdentityCommandConflict`, `create_or_get_public_identity_command()`, `get_public_identity_command()`, and conditional `advance_public_identity_command()` records keyed by a normalized-email digest and immutable issuer/subject/user/role fingerprint.
- `src/stoa/services/public_identity_service.py`: `start_or_resume_public_registration()`, `confirm_and_reconcile_public_identity()`, and `resolve_public_access_token()` orchestration.
- `tests/test_public_identity_lifecycle.py`: offline end-to-end student/parent lifecycle, conflict, retry, duplicate-email, subject mismatch, and revoked-status proof.

<must_haves>
Truths: confirmed public accounts have exactly one canonical issuer-subject binding; login/refresh select only the verified subject's active profile; retries never broaden or re-point identity. Artifacts: the public identity repository/service and lifecycle test module named above. Key links: provider `UserSub` flows into the same binding read by `resolve_actor`, and every token response passes the Plan 02 verifier before profile selection.
</must_haves>

<threat_model>
Assets: Cognito subjects, stable local user IDs, student/parent profiles, access tokens, and account lifecycle state. Trust boundaries: public email/password/code input to Cognito; provider `UserSub` and verified token to local command/binding/profile; eventually consistent email indexes to security decisions. Threats: an attacker binds their subject to another profile, duplicate emails select an arbitrary profile, confirmation activates a partial identity, retry re-points a binding, or a revoked profile receives tokens/profile data. Mitigations: immutable issuer/subject/user/role command fingerprint, conditional unique binding, provider-subject equality on every resume, deny-first non-active profile until group/binding/verification converge, access-token cryptographic verification followed by `resolve_actor`, and no email GSI in login/refresh identity selection. Verification: positive register-confirm-Actor flows for both roles; negative duplicate/mismatch/conflict/revoked cases; partial-failure mutation/replay tests; email-fallback mutation canaries. Residual risk: Cognito and DynamoDB are not transactional, so interrupted commands may remain non-active until retry; they never authorize. Cross-phase deferrals: Phase 474 owns strict Settings/global-suite fixtures, Phase 475 owns multi-write relationship and teacher-takeover transactions, and live Cognito evidence remains an explicit optional sandbox checkpoint without production mutation. No phase-owned High/Critical threat may remain unresolved.
</threat_model>
