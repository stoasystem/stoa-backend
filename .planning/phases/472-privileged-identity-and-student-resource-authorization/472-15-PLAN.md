# Plan 472-15: Safe structured public Cognito error boundary

<objective>
Close G-05 by routing every public Cognito failure through one safe structured boundary, keeping provider diagnostics internal/redacted and preserving simple actionable client behavior.
</objective>

## Artifacts this phase produces

- `src/stoa/security/public_auth_errors.py`: `PublicAuthOperation`, `NormalizedPublicAuthFailure`, `normalize_cognito_failure()`, and `public_auth_error_response()`.
- `tests/test_public_auth_error_boundary.py`: parameterized endpoint/known/unknown/canary coverage for register, login, verification resend/confirm, forgot/reset password, refresh, and logout.

<must_haves>
Truths: all public provider failures have one stable structured response and action, raw provider details never escape, and retries stay bounded. Artifacts: the public auth error module/test named above. Key link: every Cognito-backed public route uses the same normalizer and generated client registry.
</must_haves>

<threat_model>
Assets: account existence, provider topology/status, client recovery behavior, and correlation evidence. Trust boundaries: Cognito `ClientError` code/message to public HTTP response; internal structured error to generated web/mobile contract. Threats: provider codes/messages fingerprint accounts/configuration, inconsistent shapes break safe client actions, retries replay writes, unknown exceptions leak through `HTTPException.detail`, or redaction logs copy canaries. Mitigations: closed operation-aware normalization table, top-level allowlisted safe response, normalized correlation ID, generic dependency failure for unknown codes, bounded Retry-After only where allowed, exhaustive generated action registry, provider diagnostics represented only by redacted internal category/hash, and parameterized response/log canaries. Verification: every endpoint × unknown failure plus known credential/verification/reset/rate-limit controls, JSON exact-field checks, registry completeness, and byte-stable generation. Residual risk: provider outages reduce availability and some public existence-hiding flows intentionally return accepted; they disclose no internal cause. Cross-phase deferrals: Phase 478 owns actual UI rendering from the generated contract; Phase 474 owns global Settings fixtures; no live provider call or production mutation is authorized. No phase-owned High/Critical threat may remain unresolved.
</threat_model>
