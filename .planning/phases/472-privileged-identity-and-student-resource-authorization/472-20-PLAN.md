# Plan 472-20: Password recovery anti-enumeration equivalence

<objective>
Close WR-02 by making password-recovery initiation and invalid reset outcomes indistinguishable across account existence and lifecycle state, while retaining structured internal actions and simple safe client guidance.
</objective>

## Artifacts this phase produces

- Operation-aware recovery normalization rules with one public initiation projection and one invalid-reset failure class.
- Known/unknown/disabled/privileged equivalence matrices across registered HTTP endpoints and generated client actions.

<must_haves>
Truths: initiation never exposes delivery/account state; invalid reset never exposes local/provider state; client actions remain safe/actionable. Artifacts: public auth error boundary and endpoint tests. Key link: recovery provider outcomes enter the closed normalizer and public projection without an email-profile security lookup.
</must_haves>

<threat_model>
ASVS L1 assets: account existence, role, enabled/disabled status, recovery delivery destination, provider diagnostics, reset codes, and client behavior. Trust boundaries: public email/code/password to local lookup/provider call; provider account-state errors to HTTP response; structured code to generated UI action. Threats: delivery metadata confirms an account, disabled/privileged profiles produce distinct codes, unknown reset returns legacy `detail`, provider code/message leaks, or safe copy reveals internal lifecycle. Mitigations: remove local profile preclassification, uniform provider invocation/projection, omit delivery, operation-specific closed mapping, exact response keys/status/message class, server correlation, provider-free generated copy, and no automatic write replay. Verification: full known/unknown/disabled/privileged matrices with identical outputs, mutation/canary assertions, and valid recovery positive controls. Residual risk: network timing and provider throttling can affect availability; no account-specific state is intentionally projected. Phase 478 owns UI rendering, Phase 474 owns strict Settings fixtures, and no live provider/production mutation is required.
</threat_model>
