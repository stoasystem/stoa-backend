# Plan 472-14: Durable authorization decisions and bounded probe evidence

<objective>
Close G-04 by wiring one request-scoped correlation ID and durable audit sink through every central-policy call path, persisting required denials/sensitive allows, aggregating probes without sensitive data, and failing closed on audit outage.
</objective>

## Artifacts this phase produces

- `src/stoa/security/request_correlation.py`: `get_request_correlation_id()` creates/caches one canonical server request ID and attaches it to the response.
- `tests/test_authorization_audit.py`: call-graph coverage, persistence, redaction, aggregation, idempotency, outage, and handler-effect canaries.
- Existing `security_audit_repo.py` gains `AuthorizationAuditSink`, `DynamoAuthorizationAuditSink`, `persist_authorization_decision()`, and `aggregate_authorization_probe()`.

<must_haves>
Truths: required decisions produce durable redacted evidence, probes aggregate within bounds, and outage never permits effects. Artifacts: the request-correlation dependency and audit test module. Key link: the same request ID reaches policy event, persistence/aggregation, and the safe error/response before a mandatory allow returns.
</must_haves>

<threat_model>
Assets: decisions, student-resource evidence, correlation IDs, audit-key material, and enumeration detection. Trust boundaries: HTTP request to canonical ID; resource identity to keyed fingerprint; every policy/operator/admin decision entry to audit repository; audit availability to handler execution; untrusted identifiers to aggregate keys. Threats: direct evaluate/operator/admin paths omit evidence, two different resources in one request collapse, replay double-counts, raw identifiers leak through event keys, missing/rotated keys weaken authorization, probe floods create rows, or audit outage allows. Mitigations: server-generated per-request UUID, explicit correlation/sink propagation through every decision entry, HMAC resource fingerprints under a dedicated versioned keyring, allowlisted events, deterministic per-request/per-resource decision IDs, prior-key replay lookup during rotation, fixed bounded aggregate windows/counters/TTL, evidence-before-handler, and explicit outage semantics. Verification covers `authorize_and_resolve`, every direct `AuthorizationPolicy.evaluate`, `operator_capability_permits`, and admin decisions; same-request different-resource separation; same-decision replay; distinct-request aggregation; safe body/header/audit equality; sink/key outage; and effect sentinels. Residual risk: denial evidence outage degrades detection but remains a denial; mandatory allows become safe 503. Phase 480 owns broader alerts; Phase 474/475 do not own this local evidence boundary. No phase-owned High/Critical threat may remain unresolved.
</threat_model>

  <action>Create `request_correlation.py` with FastAPI dependency `get_request_correlation_id(request, response)`. On first use generate a server UUIDv4, pass it through `normalize_correlation_id()`, cache it at `request.state.stoa_correlation_id`, and set response header `X-Correlation-ID`; later dependencies in the same request return it. Do not use `"authorization"`, raw resource IDs, or a caller-reusable inbound header as canonical correlation.

Add a dedicated authorization-audit HMAC keyring to `Settings`: an active key ID and secret plus optional previous key IDs/secrets for bounded rotation lookup. Production requires a non-placeholder active key; tests inject fixed keys. Expose an overrideable `get_authorization_audit_sink()` in `deps.py` that constructs `DynamoAuthorizationAuditSink` only from validated settings. Key material is never logged, persisted, returned, or included in event IDs. A missing/invalid active key is an evidence outage: denials remain denials; any allow requiring durable evidence becomes `authorization_temporarily_unavailable`; optional owner-self allows may proceed only under the existing `requires_durable_evidence()` rule. Rotation writes with the active key ID and, for replay detection only, computes candidate IDs with retained previous keys; old rows are immutable, and prior keys must be retained for at least the decision retry/probe TTL before removal.

Define `AuthorizationAuditSink` and `DynamoAuthorizationAuditSink`. Build canonical resource identity material from the decision's resource type, exact resource ID, student/owner boundary, and scope discriminator using length-delimited canonical encoding; compute `resource_fingerprint = HMAC-SHA256(audit_key, material)`. Persist only the fingerprint and non-secret key ID, never the material or any raw resource/student/owner ID. Compute the immutable event ID from a length-delimited normalized tuple of `correlation_id`, `actor_id`, `policy_version`, `resource_type`, `action`, `purpose`, `result`, `decision_kind`, `resource_fingerprint`, and key ID. This makes two different resources with otherwise identical decisions in one request distinct while the exact same decision replay is idempotent. Store the same correlation ID, policy result/type (`authorization_denied` or `authorization_sensitive_allowed`), policy version, timestamp, fingerprint, and key ID on an explicit allowlist.

Create one decision-recording gateway used after policy/operator evaluation and before any return/raise. Add correlation and sink dependencies to every generated dependency in `route_authorization.py`; explicitly convert all three direct `AuthorizationPolicy.evaluate` branches and the `operator_capability_permits` reviewer branch to the gateway, not only `authorize_and_resolve`. Thread the same values through helper signatures/callers in `parents.py`, `practice.py`, and `adaptive.py`. Inject the same correlation and sink into `admin_operation()` and record its early role/grant denial, target-resolution denial, exact capability denial, and successful capability decision. If the boolean operator helper cannot produce the required dimensions, add an internal decision-returning wrapper and keep the boolean public helper as a compatibility projection; no production branch may bypass recording.

Centralize `requires_durable_evidence()` for parent/teacher student access, all admin support/safety/content/report reads and other admin capability decisions, break-glass, export/external-send, and purpose/capability-sensitive allows; owner self-service may omit it. Every deny attempts persistence before raising. Ensure every error uses the same correlation ID. Add AST and registered-route completeness tests that enumerate `authorize_and_resolve`, direct `.evaluate`, `operator_capability_permits`, and `admin_operation` decision branches and fail when any reaches return/raise without the sink gateway.</action>
