# Plan 472-01: Security contracts and Wave 0 verification harness

<objective>
Create the stable security vocabulary, safe failure/event boundary, injected test doubles, and collectible red-test surfaces that every later plan must implement. This plan does not add privilege or connect to real AWS.
</objective>

## Artifacts this phase produces

- `src/stoa/security/errors.py`: `SecurityErrorCode`, `SecurityDecisionError`, safe response serializer, correlation-ID normalization, and status/code mapping.
- `src/stoa/security/events.py`: allowlisted `SecurityEvent` projection and canary-redaction helpers.
- `src/stoa/security/identity.py`: immutable `Actor`, `AccountStatus`, canonical one-role model, identity/grant repository protocols.
- `src/stoa/security/authorization.py`: `AuthorizationSpec`, resource/action/purpose enums, `PolicyDecision`, and repository protocols (evaluation arrives in 472-05).
- `src/stoa/security/client_error_actions.py` plus generated `docs/security/client-error-actions.json`: one versioned, machine-readable error-to-message/action/retry contract that later web/mobile clients can consume without learning internal authorization structure.
- `scripts/generate_client_error_actions.py` and `tests/test_client_error_actions.py`: deterministic generation, schema/coverage checks, response-header semantics, and a reference client decision interpreter. Actual UI rendering and application integration remain Phase 478.
- `tests/security/conftest.py`: frozen clock, two RSA/JWKS keysets, fake async JWKS transport, fake Cognito call recorder, Actor/resource/binding/assignment/grant builders, failing-repository doubles.
- Eight named Wave 0 test modules from `472-VALIDATION.md`; tests are real assertions, not skipped placeholders. Later-behavior modules may initially be red but must collect.

## Locked decision coverage

- D-26: enforce the stable 401/403/409/503 taxonomy.
- D-27: distinguish safe 403 from existence-hiding 404.
- D-28: expose only `code`, safe `message`, and `correlationId`.
- D-29: preserve actionable UI mappings without internal detail.
- D-31: encode bounded retry guidance by error/action class.
- D-32: emit only the allowlisted redacted security-event fields.
- D-33: authorization fact-store uncertainty returns safe 503.

<threat_model>
Assets: privileged identities, student-resource existence, tokens, security evidence, and client recovery behavior. Trust boundaries: HTTP request to FastAPI, provider/repository doubles to domain contracts, internal event to client response, generated shared contract to later clients. Abuse cases: unsafe exception text reaches clients; blanket-deny passes a denial-only suite; token/email/content/object-key canaries enter audit output; ambiguous roles are normalized; clients retry a denied/non-idempotent operation or loop token refresh. Mitigations: closed enum/code taxonomy, immutable Actor with exactly one canonical role, allowlisted event serializer, positive controls in matrix cases, exhaustive generated client-action mapping with bounded retry invariants, collection tests for every Wave 0 module. Validation references: T-472-01 through T-472-09, especially T-472-07/T-472-09. ASVS L1 High/Critical failures block completion.
</threat_model>
