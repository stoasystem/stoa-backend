# Plan 472-19: Typed admin body-target authorization and audit identity

<objective>
Close WR-01 by passing validated body-only administrator targets explicitly into the central capability gateway, using one canonical target for scope and audit evidence, and enforcing executable inventory coverage for every affected route.
</objective>

## Artifacts this phase produces

- A shared typed admin-target authorization contract and route dependency factory with explicit scalar, collection, and reference-only cardinality; arbitrary raw JSON is never interpreted as authority.
- Registered-route inventory checks and endpoint/audit regressions for every body-declared admin target family, including later-element denial and audit failure.

<must_haves>
Truths: every body target is authorized before effects, wrong or partial scopes deny the whole command, and distinct targets have distinct evidence. Artifacts: admin authorization/inventory modules and focused tests. Key link: each typed body target coordinate becomes both `ResourceRef.resource_id/student_id` input and the HMAC fingerprint material persisted by the existing recording gateway; a collection digest may correlate the command but never substitutes for per-target decisions/evidence.
</must_haves>

<threat_model>
ASVS L1 assets: parent/student bindings, curriculum versions, report/recovery targets, capability scopes, handler effects, and redacted authorization evidence. Trust boundaries: untrusted JSON to Pydantic validation; validated body fields to authorization targets; each target coordinate to grant matching and HMAC evidence; all-of dependency success to handler execution. Threats: body target silently defaults global, only the first collection element is checked, mixed students collapse into one actor/global target, partial authorization still mutates, a later audit write fails after an earlier allow, global audit identities collapse unrelated resources, delimiter ambiguity aliases targets, evidence-only dictionaries are mistaken for authority, raw-body guessing trusts undeclared fields, or missing typed coverage drifts in a new route. Mitigations: explicit per-route cardinality/provider classification, declared-key allowlist, bounded unique canonical ResourceRef sets, per-ref all-of decision and durable allow evidence before every effect, canonical length-delimited key/value coordinates, recursive registered-route coverage, exact allow/deny matrices, and later-target outage sentinels. Verification: every route where Pydantic body aliases intersect `AdminRoutePolicy.target_keys`, including parent binding, curriculum, report bulk/recovery/handoff/governance families. Residual risk: truly global admin operations remain deliberately global and are separately classified. Phase 474 Settings fixtures and Phase 475 teacher takeover remain out of scope; no production/provider mutation is required.
</threat_model>

## Locked route cardinality and target classification

The executor must encode this classification in route policy/provider metadata and prove it against registered FastAPI dependencies; it is not documentation-only:

- **Scalar target** — parent/student binding repair, one curriculum lesson/version command, and report routes addressed by one path/query/body report/job/delivery tuple yield exactly one canonical `ResourceRef`. Missing required coordinate parts, conflicting path/query/body values, or more than one logical target is a safe validation denial. Optional fields may enrich that same coordinate but may not create an accidental fallback target.
- **Explicit collection** — `BulkReportResendRequest.reports` (1–25 report tuples), `SupportHandoffPackageRequest.recovery_job_ids` (0–5 job references, required non-empty only when the selected destination/action consumes recovery sections), and audit-retention/immutable-evidence/legal-hold/governance `references` (route-declared 0/1–10 semantics) yield a bounded, deterministic, unique set of canonical `ResourceRef`s. Bulk resend may span students only when every report tuple is independently authorized; governance/handoff collections likewise require authority for every independently actionable job/report/student coordinate. No route may choose the first item, authorize a composite instead of its members, or replace multiple students with `actor.user_id`/`global`.
- **Resolver-backed collection** — recovery preview/create/resume commands that select reports through `filters`, `source_job_id`, results, or a preview token must use an allowlisted read-only resolver to materialize the bounded exact report target set before the command mutation. Create/resume must bind to the same canonical set sealed by the preview; changed, empty-required, expired, or over-limit resolved sets deny before job creation. The resolver may perform only target-identification reads and may not enqueue, update, send, persist a business object, or invoke the command service before all-of authorization and evidence complete.
- **Reference/evidence only** — `reason`, `operator_note`, arbitrary `release_evidence`, retention `evidence_references`, `break_glass`, provider/message metadata, report payload/content, `proposed_fields`, pagination/limit controls, fixture names, expected artifact versions, and similar nested descriptive dictionaries are never scanned as authorization targets unless the route policy explicitly promotes a typed field. In `SupportHandoffFixtureReference`, declared `parent_id`/`student_id`/`week_start` form a target tuple when the fixture causes target/report retrieval; `fixture_name` and expected-version data remain evidence/reference-only.

For every collection, validate element shape and required tuple completeness, canonicalize each element with typed length-prefixed field encoding, reject canonical duplicates rather than silently processing them twice, sort the unique refs deterministically for decision/evidence order, and enforce the route's pre-declared maximum. Mixed-student collections are not collapsed: they succeed only if all refs allow. The gateway must evaluate every ref and durably record the redacted allow for every ref before releasing the typed command. Any malformed/conflicting/empty-required/over-limit/duplicate set, any denied ref, or any per-target evidence failure fails the entire command with zero handler/service mutation. A length-prefixed collection correlation fingerprint may be recorded in addition to, but never instead of, the per-target fingerprints.
