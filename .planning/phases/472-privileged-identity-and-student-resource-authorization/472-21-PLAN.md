# Plan 472-21: Strong production audit HMAC keyring validation

<objective>
Close WR-04 by enforcing uniformly strong, unique, safely parsed authorization-audit HMAC key material at configuration and sink construction boundaries, with rotation-positive and secret-redaction tests.
</objective>

## Artifacts this phase produces

- One canonical audit key/keyring validator used by production `Settings` and defensive sink construction.
- Startup/configuration tests for weak material, normalized duplicates, valid rotation, redacted failures, and development behavior.

<must_haves>
Truths: weak/default/duplicate material cannot start production; all retained keys meet the active-key standard; errors contain no secrets. Artifacts: Settings/sink validation and tests. Key link: the same normalized byte material used by HMAC fingerprinting is what the validator measures and compares.
</must_haves>

<threat_model>
ASVS L1 assets: actor/resource privacy in durable audit rows, HMAC key confidentiality, rotation history, startup availability, and replay recognition. Trust boundaries: environment strings to decoded key bytes; Settings validation to cached dependency/sink; active/previous keyring to HMAC fingerprints and retained-key replay checks. Threats: one-character key enables offline guessing, repeated/predictable material looks non-default, malformed encoding truncates entropy, duplicate active/previous keys defeat rotation identity, whitespace-normalized ID collision selects ambiguity, or validation errors/logs expose secrets. Mitigations: canonical parsing, minimum 256-bit material, conservative unpredictability/placeholder checks, uniform active/previous validation, normalized unique IDs, unique key digests, fail-fast production startup, defensive sink validation, and secret-free error categories. Verification: short, repeated, common placeholder, malformed encoded, duplicate ID/material, valid active/retained rotation, replay, and development-only fixtures. Residual risk: software cannot prove true generation entropy; operator setup must use a cryptographic secret generator and secret manager. Phase 474 owns its strict Settings fixture modernization; this plan must not edit or reassign those 23 fixtures. No production secret or external service is accessed.
</threat_model>
