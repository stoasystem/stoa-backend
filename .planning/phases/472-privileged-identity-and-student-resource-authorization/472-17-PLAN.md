# Plan 472-17: Proof-bound public identity resume and resend convergence

<objective>
Close CR-01 and WR-03 by making existing-account registration and verification resend resumable only through the pre-existing immutable public identity command, with proof-bound subject matching and no email-index identity activation.
</objective>

## Artifacts this phase produces

- A command-first public registration resume boundary in `public_identity_service.py`; no new authority store is introduced.
- Command-aware resend regression coverage for provider-only, privileged, duplicate-email, subject-mismatch, already-confirmed, and partial-failure cases.

<must_haves>
Truths: an existing provider account is never adopted without a matching command; legitimate interrupted commands resume safely; resend never activates an email-GSI-selected row. Artifacts: the service and lifecycle/endpoint tests named above. Key links: both register-existing and resend-already-confirmed paths load the immutable command before provider subject comparison or mutation and converge through the same activation service.
</must_haves>

<threat_model>
ASVS L1 assets: provider subjects, stable local user IDs, one-role account ownership, parent/student relationship inputs, provider groups, and active profiles. Trust boundaries: unauthenticated email/role/profile input to provider lookup; email-keyed command lookup to immutable issuer/subject/user/role fingerprint; provider confirmation state to local activation. Threats: account adoption after `UsernameExistsException`, privileged subject denial/corruption through a second group, altered retry profile data, duplicate-email row activation, and partial convergence presented as active. Mitigations: command-before-provider-lookup precondition, exact fingerprint comparison, zero-mutation failure, command-user profile lookup, one canonical `student|parent` group, shared `confirm_and_reconcile_public_identity()`, deny-first intermediate states, and structured safe existing-account/retry actions. Verification: HTTP mutation recorders and repository/provider doubles for provider-only public, privileged, legitimate interrupted, duplicate-email, mismatch, and retry cases. Residual risk: interrupted commands may remain unavailable until a legitimate retry; they cannot authorize. Cross-phase deferrals: teacher takeover atomicity remains Phase 475 and strict production Settings fixtures remain Phase 474. No AWS, network, provider sandbox, or production mutation is required.
</threat_model>
