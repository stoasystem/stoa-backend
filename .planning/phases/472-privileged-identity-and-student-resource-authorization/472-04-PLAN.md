# Plan 472-04: Versioned capabilities and privileged identity lifecycle

<objective>
Implement no-privilege teacher application/approval/invitation activation, capability-bounded routine admin lifecycle, immediate suspension/revocation, and a narrowed bootstrap/disaster-recovery script.
</objective>

## Artifacts this phase produces

- Versioned `capability_repo` and append-only redacted `security_audit_repo`.
- Immutable teacher application versions, exact-version reviews, digest-only invitations, conditional single-use activation commands.
- Public application plus capability-guarded reviewer/admin endpoints in `teacher_applications.py` and `admin.py`.
- Authenticated routine admin lifecycle service guarded by `admin_identity_manager`.
- Hardened bootstrap script requiring explicit purpose/incident reason and identity binding.

## Locked decision coverage

- D-01: public application is non-privileged; activation requires the bound invitation.
- D-02: offline review requires active `teacher_identity_reviewer`.
- D-03: application versions are immutable and reviewed exactly.
- D-04: approval alone does not elevate; every lifecycle transition is evidenced.
- D-05: suspension/revocation denies immediately and triggers provider defenses.
- D-06: bootstrap script is recovery-only; routine admin creation is authenticated/capability-bound.
- D-07: no root/admin omnipotent request-path bypass exists.
- D-08: only approved exact matches migrate; anomalies suspend pending review.
- D-09: full applications are reviewer-only and audit records copy no sensitive content.
- D-23: current versioned local grants are the only capability authority.
- D-24: reconciliation may tighten automatically but elevation/restoration/grant needs approval.

<threat_model>
Assets: teacher/admin role activation, capability grants, application data, lifecycle evidence. Trust boundaries: public applicant to no-access record; reviewer/admin Actor to durable transition; DynamoDB command to Cognito mutation. Abuse cases: approval directly elevates, invitation replay/race creates duplicate teacher, wrong email consumes token, partial provider failure leaves active local privilege, broad admin/root grants itself authority, sensitive application/audit content leaks. Mitigations: immutable versions, exact reviewer capability, approval-only invitation issuance, digest-bound expiry/same verified email, conditional consume and resumable idempotent command, deny-first non-active state until reconciliation, versioned grants, bootstrap isolation, bounded projections. Validation: T-472-04, T-472-05, T-472-07.
</threat_model>
