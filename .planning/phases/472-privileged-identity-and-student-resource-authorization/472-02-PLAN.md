# Plan 472-02: Access-token verification and explicit identity boundary

<objective>
Replace the mutable, fallback-heavy authentication dependency with issuer/client/use-bound token verification and explicit `(issuer, subject) -> user_id` resolution that intersects one Cognito role with fresh local state.
</objective>

## Artifacts this phase produces

- `security/jwks.py`: injected async issuer-keyed/kid-keyed bounded JWKS cache with timeout and single refresh.
- `security/tokens.py`: verified access-token result and stable redacted failures.
- `db/repositories/identity_repo.py`: conditional unique binding plus inventory-only reverse records.
- `security/identity.py`: fresh account/grant resolution and identity-conflict/recovery behavior.
- `deps.py`: thin `get_verified_token`, `get_actor`, and temporary read-only legacy adapter; no provider mutation.

## Locked decision coverage

- D-18: privilege is verified Cognito/local intersection with no request-time repair.
- D-19: resolve only unique `(issuer, sub) -> user_id`; never email fallback.
- D-20: require exactly one matching role/group and never select highest privilege.
- D-21: only active status authorizes protected access.
- D-25: conflicted identities receive only the safe recovery surface.
- D-30: known-key bounded caching and one unknown-kid refresh remain fail closed.

<threat_model>
Assets: access tokens, signing trust, stable business identity, current privilege state. Trust boundaries: untrusted JWT to configured issuer/key/client; provider claims to local binding/profile/grants. Abuse cases: wrong-pool/client/ID token accepted, unknown key bypass, multi-group highest-role selection, email fallback binds attacker, stale local grants continue after revocation, provider outage fails open. Mitigations: exact configured issuer and client allowlist, access-token-only verification, bounded issuer-isolated JWKS, one forced unknown-kid refresh, unique binding, exact one canonical group matching fresh active local role, no cross-request Actor/allow cache, safe 503. Validation: T-472-02, T-472-03, T-472-05, T-472-07.
</threat_model>
