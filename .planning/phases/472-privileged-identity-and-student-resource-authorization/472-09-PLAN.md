# Plan 472-09: Admin and operator route capability migration

<objective>
Replace the `/admin` prefix and `require_role("admin")` as authority with an explicit capability/purpose/resource classification for every admin-prefixed route registered in `main.py`, and enforce Actor-derived ownership/central policy for user notification, event and push-token resources.
</objective>

## Artifacts this phase produces

- `tests/test_admin_authorization.py`: generated table of every `admin.py` route and required capability/purpose/resource/action with positive and denial controls.
- Capability-scoped dependencies on all admin routes; no blanket student-content access.
- Safe support metadata projection and incident-only read-only break-glass enforcement.
- Explicit global-operator classifications for non-student curriculum/BI/readiness routes rather than implicit admin role.
- Actor-owned notification list/preferences/digest/event/push-token routes, plus distinct exact capabilities for admin notification inspection and delivery-health inspection.

<threat_model>
Assets: all user/student/parent/report/billing/usage/notification data, push-token references, and privileged operations. Trust boundaries: admin coarse role to versioned capability; Actor to notification recipient ownership; path/body IDs to resource scope; export/send/mutation to elevated action. Abuse cases: ordinary admin reads student content or operational events, user marks another recipient's event, user revokes another device's token, support lookup leaks messages/answers, report/export route bypasses child policy, break-glass mutates/sends/exports, privilege endpoint self-elevates, separately registered admin router escapes the `admin.py` table. Mitigations: exhaustive main-app route table, exact capability/action/purpose, Actor-owned notification resolution, bounded projection, read-only incident constraints, no implicit role bypass, positive and negative route tests, global inventory fail on role-only sensitive dependency. Validation: T-472-05, T-472-06, T-472-07, T-472-08.
</threat_model>
