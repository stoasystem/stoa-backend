# Phase 12 Plan 01: Backend Report Flow Verification

## Goal

Strengthen backend tests so the weekly report flow is verified from scheduled orchestration through aggregation, generation/fallback, storage, email failure handling, idempotency, and parent API state responses.

## Tasks

### 1. Add backend flow verification tests

Files:

- `tests/test_report_flow.py`

Work:

- Add an integration-style test for one scheduled pair that exercises:
  - atomic generation claim,
  - weekly aggregation from faked student activity,
  - deterministic fallback generation when Bedrock fails,
  - JSON/HTML artifact writes,
  - DynamoDB metadata creation,
  - SES delivery status update.
- Add an email-failure variant or assertion that confirms generated report availability is preserved as `email_failed`.
- Add an S3-failure ordering test proving metadata and email are not written when artifacts fail.

Acceptance:

- The flow test proves the services connect without live AWS.
- Failure ordering is explicit and protects report durability semantics.

### 2. Cover remaining backend edge cases

Files:

- `tests/test_report_service.py`
- `tests/test_weekly_reports_job.py`
- `tests/test_parent_children.py`

Work:

- Add aggregation pagination coverage for student questions if missing.
- Add Zurich/EventBridge edge coverage if current tests leave a meaningful gap.
- Ensure parent API generated, missing, pending, failed, and email-failed states remain covered and parent-safe.

Acceptance:

- `TEST-01` through `TEST-06` each have direct test evidence.
- No test depends on external network, live AWS, or real DynamoDB/S3/SES.

### 3. Run focused backend verification

Commands:

- `uv run --extra dev pytest tests/test_report_flow.py tests/test_weekly_reports_job.py tests/test_report_service.py tests/test_parent_children.py -q`
- `uv run --extra dev ruff check src/stoa/jobs/weekly_reports.py src/stoa/services/report_service.py src/stoa/db/repositories/report_repo.py src/stoa/routers/parents.py tests/test_report_flow.py tests/test_weekly_reports_job.py tests/test_report_service.py tests/test_parent_children.py`

Acceptance:

- Focused report backend tests pass.
- Ruff passes for touched backend modules and tests.

## Verification Criteria

- `TEST-01` is satisfied when tests cover Zurich week-window calculation and idempotent scheduled generation.
- `TEST-02` is satisfied when tests cover empty and mixed weekly aggregation plus Bedrock parser and fallback behavior.
- `TEST-03` is satisfied when tests cover DynamoDB metadata creation and S3 JSON/HTML artifact writes.
- `TEST-04` is satisfied when tests cover SES failure marking `email_failed` after storage.
- `TEST-05` is satisfied when tests cover parent endpoint generated, missing, pending, failed, and email-failed states without raw generation error exposure.
- `TEST-06` is satisfied when the full backend report flow is covered with faked AWS clients and repository calls.

## Risks

- Existing tests already cover much of this phase; avoid broad rewrites that make the suite less focused.
- Integration-style fakes can become brittle if they duplicate implementation internals too closely; keep assertions tied to required side effects and user-visible status.
