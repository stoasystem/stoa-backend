## Phase Decision Map

- D-01: Use local DynamoDB parent profile `user_id` as the canonical parent ownership identifier.
- D-02: Resolve authenticated parent identity with `user_repo.get_user(user["sub"])`, then Cognito email, then `user_repo.get_user_by_email(email)`.
- D-03: Require the resolved profile role to be `parent`; do not admit admin, student, teacher, or tutor through `/parents/me/...`.
- D-04: Do not trust client-supplied parent IDs in the normal parent portal flow.
- D-05: Implement child lookup by scanning student profile items where `role == "student"` and `parent_id == resolved_parent_user_id`.
- D-06: Paginate the scan across all DynamoDB pages.
- D-07: Return `{ "items": [...] }`, including `{ "items": [] }` for no linked children.
- D-08: Child list items include `id`, `userId`, `name`, `email`, `grade`, `subjects`, and `relationship`.
- D-09: Keep legacy `/parents/{parent_id}/children` and `/parents/{parent_id}/reports/{week}` available.
- D-10: If legacy parent routes are touched, compare requested `parent_id` with the resolved local parent profile ID rather than raw JWT `sub`.
- D-11: Keep admin compatibility for legacy path-ID routes only if it already exists; do not allow admin through `/parents/me/...`.
- D-12: Keep helper functions route-local in `src/stoa/routers/parents.py` unless a small repository helper clearly reduces duplication.
- D-13: Use existing FastAPI/Pydantic patterns from the current router.
- D-14: Add minimal tests for confidence; broader test/data coverage remains Phase 5.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cognito JWT to FastAPI dependency | Untrusted bearer token claims become backend authorization input after JWT validation. |
| FastAPI route to DynamoDB scan | Authenticated route code reads profile and child records from DynamoDB. |
| DynamoDB records to frontend response | Stored user profile fields become parent-visible child list fields. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01-S | Spoofing | `_resolve_parent_profile` | mitigate | Resolve Cognito claims to a local DynamoDB parent profile through direct lookup and Cognito email fallback before using ownership IDs. |
| T-02-01-T | Tampering | `GET /parents/me/children` | mitigate | Do not accept client-supplied parent IDs in the normal flow; derive `parent_user_id` only from the resolved profile. |
| T-02-01-I | Information disclosure | `_scan_children_for_parent` | mitigate | Filter scan by `parent_id == resolved.parent_user_id` and `role == "student"`; reject non-parent roles through `require_role("parent")` and resolved profile role check. |
| T-02-01-D | Denial of service | DynamoDB scan pagination | accept | Phase 1 accepted scan-based child lookup for this milestone; pagination avoids incomplete results, and a dedicated GSI remains outside this phase unless implementation evidence proves the scan path unusable. |
| T-02-01-E | Elevation of privilege | `/parents/me/...` | mitigate | Admin, student, teacher, and tutor roles are not accepted by `require_role("parent")`; admin compatibility is limited to legacy path-ID routes in Plan 02. |
</threat_model>
