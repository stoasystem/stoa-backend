# Plan 158-01: Implement Direct Refund Execution And Finance Handoff

<objective>
Add controlled admin direct refund execution with idempotency, audit persistence, billing projection updates, and finance handoff evidence.
</objective>

<tasks>

## Task 1: Add refund execution gate and request contract

Type: code

Files:
- `src/stoa/config.py`
- `src/stoa/routers/admin.py`

Action:
- Add `stripe_refunds_enabled`, default false.
- Add an admin refund request model requiring amount, operator reason, and idempotency key.
- Add `POST /admin/subscriptions/billing/{parent_id}/refunds`.

Verify:
- Route uses existing admin role dependency.

Acceptance criteria:
- Refund mutation cannot run without explicit gate and admin caller.

## Task 2: Implement refund service flow

Type: code

Files:
- `src/stoa/services/subscription_service.py`

Action:
- Validate billing eligibility, provider reference, remaining refundable amount, TWINT refund window, operator reason, and idempotency key.
- Persist idempotency/audit state.
- Execute provider refund through a dedicated seam.
- Update billing invoice/refund summary/accounting handoff and billing event evidence only after provider success.

Verify:
- Provider failure leaves billing amount/refund state unchanged.

Acceptance criteria:
- Success, refusal, duplicate/idempotent replay, and provider failure paths are explicit and safe.

Acceptance criteria:
- Finance can reconcile invoice and refund details from one export row.
