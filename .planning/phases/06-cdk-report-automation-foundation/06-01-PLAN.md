# Phase 6 Plan 01: CDK Weekly Report Automation Foundation

## Goal

Define the weekly report automation infrastructure in CDK before backend generation code depends on it.

## Scope

This plan changes the infrastructure repository `/Users/zhdeng/stoa-infra` and adds only the minimal backend scheduled handler stub needed to prevent the scheduled Lambda from failing import before Phase 10 implements the full job.

## Tasks

### 1. Add report Lambda infrastructure

Files:

- `/Users/zhdeng/stoa-infra/stacks/api_stack.py`
- `/Users/zhdeng/stoa-infra/app.py`

Work:

- Pass `storage.reports_bucket` into `ApiStack`.
- Add `S3_REPORTS_BUCKET` to the existing API Lambda environment and grant the API Lambda report bucket access for later parent report artifact reads.
- Define `self.weekly_report_function` as a separate Lambda using the existing backend deployment artifact.
- Use handler `stoa.jobs.weekly_reports.handler`.
- Set report Lambda environment for DynamoDB table, reports bucket, Cognito settings, and Bedrock model ID.
- Grant report Lambda DynamoDB read/write, reports bucket read/write, Bedrock invoke, and SES send permissions.

Acceptance:

- CDK code creates a separate scheduled report Lambda and does not route scheduler events through API Gateway or Mangum.
- Report Lambda has the required environment variables and permissions.

### 2. Wire EventBridge Scheduler target and failure handling

Files:

- `/Users/zhdeng/stoa-infra/stacks/notification_stack.py`
- `/Users/zhdeng/stoa-infra/app.py`

Work:

- Add a weekly report DLQ.
- Add a method on `NotificationStack` to create the weekly report schedule once the report Lambda exists.
- Create an EventBridge Scheduler target for the report Lambda in the existing `stoa-schedules` group.
- Use Zurich timezone scheduling for Monday morning report generation.
- Configure retry policy and DLQ target through CDK.
- Grant Scheduler permission to invoke the report Lambda and send failures to the DLQ.

Acceptance:

- EventBridge Scheduler has a CDK-defined target for the weekly report Lambda.
- Retry/failure handling is defined in CDK.

### 3. Add monitoring visibility

Files:

- `/Users/zhdeng/stoa-infra/stacks/monitoring_stack.py`
- `/Users/zhdeng/stoa-infra/app.py`

Work:

- Let `MonitoringStack` accept the weekly report Lambda.
- Add a CloudWatch alarm for weekly report Lambda errors.
- Add report job invocations/errors and duration widgets to the existing dashboard.

Acceptance:

- Report job failures are visible through CloudWatch alarms/dashboard widgets.

### 4. Add safe scheduled handler stub

Files:

- `src/stoa/jobs/__init__.py`
- `src/stoa/jobs/weekly_reports.py`

Work:

- Add a minimal `handler(event, context)` entrypoint returning structured zero-counts and `status: "not_implemented"`.
- Do not implement aggregation, Bedrock generation, storage, or email in this phase.

Acceptance:

- The CDK Lambda handler string resolves to an importable backend function.
- Later phases can replace the stub behavior without changing CDK handler wiring.

### 5. Verify

Commands:

- `/Users/zhdeng/stoa-backend`: `python3 -m py_compile src/stoa/jobs/__init__.py src/stoa/jobs/weekly_reports.py`
- `/Users/zhdeng/stoa-infra`: `uv run python -m py_compile app.py stacks/api_stack.py stacks/notification_stack.py stacks/monitoring_stack.py`
- `/Users/zhdeng/stoa-infra`: `uv run cdk synth` if the local CDK CLI/runtime is available.

## Verification Criteria

- `CDK-01` satisfied by separate `weekly_report_function`.
- `CDK-02` satisfied by `S3_REPORTS_BUCKET` env and report bucket grants.
- `CDK-03` satisfied by table, Bedrock, and SES permissions.
- `CDK-04` satisfied by `CfnSchedule` target.
- `CDK-05` satisfied by Scheduler retry policy and DLQ.
- `CDK-06` satisfied by report Lambda alarm/dashboard additions.

## Risks

- CDK Python property names for `aws_scheduler.CfnSchedule` may require small corrections after synth.
- Existing GitHub Actions backend packaging must include the new `src/stoa/jobs` package; it copies all of `src/stoa`, so no deploy workflow change is expected.
- Schedule may invoke the stub before Phase 10; the stub returns clean no-op counts rather than failing.
